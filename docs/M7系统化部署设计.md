## 1. 存储防坏卡：OverlayFS 只读根文件系统

**背景**：教学现场设备不能依赖人工值守，学生直接拔电源是常态；SD 卡在断电瞬间的写入很容易导致文件系统损坏、开不了机。

**做法**：
- 开启 OverlayFS，把根文件系统锁死为只读，系统运行期间的写入全部落在内存里的 upper layer，物理断电不会损坏 SD 卡本体。
- 需要写盘的场景（比如远程改配置）不让业务 C++ 程序直接持有 remount 权限：拆一个权限最小化的独立小脚本（例如 `scripts/remount_rw_and_apply_config.sh`），专职做"remount rw → 写配置 → `sync` → remount ro"这几步。
- 业务 C++ 程序通过一条极窄的 `sudoers` 规则（`NOPASSWD: /usr/local/bin/remount_rw_and_apply_config.sh <参数>`，不是给完整 shell 或 `mount` 命令本身的免密权限）去调用这个脚本，业务进程本身不持有 remount 权限。
- 具体触发时机（MQTT 收到什么样的指令、写哪些字段）留给"运行时参数配置化"子项目单独设计，本条只定"只读系统+特权收窄"这个基础设施层面的方案。

## 2. 内存防爆：日志/缓存瘦身

**背景**：OverlayFS 下所有写入都转存到内存（tmpfs），长期运行如果日志/缓存无限增长，内存会被打爆。

**做法**：
- 修改 `/etc/systemd/journald.conf`：`Storage=volatile`（只留内存，不落盘）+ `RuntimeMaxUse=16M`（锁死上限，满额后自动滚动覆盖旧日志）。
- 业务代码不自己写 `.log` 文件，统一走 `std::cout`/`std::cerr`，交给 journald 按上面的配置做流控。
- `src/main.cpp` 的 `LogJsonPayload` 现在是每处理完一帧 MAVLink 消息就打印一次完整 JSON（`payload::ToJson(...).dump(2)`）——接入 `config.json` 里已经预留但还没接上的 `logging.level` 字段，非 `debug` 级别不做这种逐帧全量 dump。这一条跟"日志分级"子项目共用同一个验收标准：**M7 上线前，`LogJsonPayload` 的默认（非 debug）运行模式下不能是无条件逐帧打印**。

## 3. 路径脱敏：禁止相对路径

**背景**：systemd 拉起的进程默认工作目录通常是 `/` 或 `/root`，代码里如果用相对路径打开文件（比如 `open("config.json")`），会因为找不到文件直接闪退。

**做法**：`cns-rpi.service`（主程序的 systemd 单元）的 `ExecStart` 用绝对路径，或者 `WorkingDirectory=` 显式指定真机部署目录，二选一，不依赖 systemd 默认工作目录（`systemd/cellular-dialup.service` 已经是这么写的，照抄同样的模式）。

## 4. 架构解耦：网络拨号与主程序分离

**背景**：如果把网络拨号和 C++ 主业务揉进同一个服务，业务重启（比如改配置触发的重启）会连带把网络一起断开重新拨号，耗时且不稳定。

**做法**：`scripts/cellular_dialup.py` + `systemd/cellular-dialup.service` 独立成一个 oneshot 单元，跟主程序（`cns-rpi.service`）之间**不建立** `After=`/`Before=`/`Requires=` 依赖关系：

- 拨号失败/重试不阻塞主程序启动——5G 只是可选的备用/主用上网通道，不是 MAVLink/UART 链路的依赖。
- 主程序自己的 MQTT 连接判断本来就有"连不上就等下一轮循环再试"的重试逻辑（见 `docs/superpowers/specs/2026-07-09-m5-mqtt-publish-design.md` §3/§7），不需要靠 systemd 的启动顺序去保证网络先就绪。
- 两个服务互不依赖，主程序的启动时间不会被 5G 拨号的重试次数拖长。

## 5. 硬件防死锁：Systemd 软件看门狗

**背景**：现场最怕"假死"——程序死锁/硬卡死但没崩溃，进程还在，但 MQTT 彻底离线，systemd 的 `Restart=on-failure` 这类基于"进程退出"的重启策略捕捉不到这种情况。

**做法**：
- `.service` 文件加 `WatchdogSec=5s`（具体秒数留到 M7 实施阶段跟实际主循环节拍对齐）。
- C++ 主程序周期性调用喂狗。现有 `src/main.cpp` 主循环是"阻塞读 UART（100ms 超时）+ 周期性检查发送心跳"的结构（`kHeartbeatInterval` 那套节拍器），加一次周期性喂狗调用可以直接复用同一个模式。
- 不链接 `libsystemd`：往 `$NOTIFY_SOCKET` 环境变量指向的 UNIX datagram socket 发送 `"WATCHDOG=1\n"` 字符串即可，几行代码，不用为这一个调用引入整个库依赖。

## 6. 部署自动化脚本

**背景**：现场部署不接显示器，安装需要一次性跑完。

**做法**：`scripts/install_deps.sh` 只在真正需要权限的单条命令上单独加 `sudo`（切 apt 源、`apt install`），git 操作和 CMake 编译以调用者身份跑，不产生 root 所有的文件，不需要事后 `chown`，维持现有的细粒度 sudo 模式。
