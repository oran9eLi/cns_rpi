## 1. 存储防坏卡：OverlayFS 只读根文件系统

**背景**：教学现场设备不能依赖人工值守（`V1设计文档.md` §7 技术选型表已经point出这个前提），学生直接拔电源是常态；SD 卡在断电瞬间的写入很容易导致文件系统损坏、开不了机。OverlayFS 把根文件系统锁死为只读，系统运行期间的写入全部落在内存里的 upper layer，物理断电不会损坏 SD 卡本体。

**采纳**：开启 OverlayFS，这是本条的核心结论，没有异议。

**不采纳原方案里的实现方式**：原方案设想"C++ 主程序收到 MQTT 远程改配置指令时，自己执行 `mount -o remount,rw /` → 写 `config.json` → `sync` → `mount -o remount,ro /` → `exit(0)` 让 systemd 拉起自身"。这个实现方式有两个问题：

1. **权限面过大**：让业务二进制拥有 remount 根文件系统的权限，等于给一个网络输入通道（MQTT）直接开了改系统挂载状态的口子。本项目 V1 阶段 MQTT 是匿名连接、无鉴权（`V1设计文档.md` §9 "Broker 地址、端口、认证信息…V1 本地调试阶段可明文"，真机侧目前也是 `allow_anonymous true`）——局域网内任何人发一条 MQTT 消息都可能触发远程改配置流程，攻击面偏大。
2. **范围耦合**："MQTT 远程改配置"这个功能本身还没设计（对应待办的"运行时参数配置化"子项目，尚未开始），不应该在设计只读文件系统方案时顺带把一个还没设计的功能的实现细节定死。

**修正后的方案**：
- 拆一个权限最小化的独立小脚本（例如 `scripts/remount_rw_and_apply_config.sh`），专职做"remount rw → 写配置 → `sync` → remount ro"这几步，不做其他事。
- 业务 C++ 程序通过一条极窄的 `sudoers` 规则（`NOPASSWD: /usr/local/bin/remount_rw_and_apply_config.sh <参数>`，不是给完整 shell 或 `mount` 命令本身的免密权限）去调用这个脚本，业务进程本身不持有 remount 权限。
- 具体触发时机（MQTT 收到什么样的指令、写哪些字段）留给"运行时参数配置化"子项目单独设计，本条只定"只读系统+特权收窄"这个基础设施层面的方案。

## 2. 内存防爆：日志/缓存瘦身

**背景**：OverlayFS 下所有写入都转存到内存（tmpfs），长期运行如果日志/缓存无限增长，内存会被打爆。

**系统层面**（采纳，无修正）：
- 修改 `/etc/systemd/journald.conf`：`Storage=volatile`（只留内存，不落盘）+ `RuntimeMaxUse=16M`（锁死上限，满额后自动滚动覆盖旧日志）。

**C++ 代码层面**（采纳方向，但指出一个已知的现存缺口）：
- 业务代码不应该自己写 `.log` 文件，统一走 `std::cout`/`std::cerr`，交给 journald 按上面的配置做流控——**不建议** `StandardOutput=null` 直接丢弃：现场设备出问题时如果什么日志都拿不到，远程排查会非常困难，只用 journald + `RuntimeMaxUse` 这一套即可，没必要连日志都不留。
- **已知缺口，正好是本条要解决的**：现在 `src/main.cpp` 的 `LogJsonPayload` 是**每处理完一帧 MAVLink 消息就打印一次完整 JSON**（`payload::ToJson(...).dump(2)`），这是 M4 阶段为了真机联调时人工核对字段可读性特意做的调试打印，不是为生产环境设计的——一旦接上 journald 常跑，这个打印频率在 tmpfs 下会持续占用内存，不容忽视。
- **对应待办**：这正是"运行时参数配置化"之后、"日志分级"子项目要解决的范围——把这条打印挂到 `config.json` 里已经预留但还没接上的 `logging.level` 字段后面，非 `debug` 级别就不打印这种逐帧全量 dump。本条给那个子项目多定一条明确的验收标准：**M7 上线前，`LogJsonPayload` 的默认（非 debug）运行模式下不能是无条件逐帧打印**。

## 3. 路径脱敏：禁止相对路径

**背景**：systemd 拉起的进程默认工作目录通常是 `/` 或 `/root`，代码里如果用相对路径打开文件（比如 `open("config.json")`），会因为找不到文件直接闪退。

**现状**：本仓库已经在按这个原则做——`systemd/cellular-dialup.service`（2026-07-09 5G拨号自动化落地时新增，目前仓库里唯一的 systemd 单元）的 `ExecStart` 用的是真机实际部署的绝对路径（`/home/dcdw/cns_rpi/scripts/cellular_dialup.py /home/dcdw/cns_rpi/config/config.json`），不是相对路径或占位路径。

**待办**：M7 设计并落地 `cns-rpi.service`（主程序的 systemd 单元）时，照抄同样的模式——`ExecStart` 用绝对路径，或者 `WorkingDirectory=` 显式指定真机部署目录，二选一，不依赖 systemd 默认工作目录。

## 4. 架构解耦：网络拨号与主程序分离

**背景**：如果把网络拨号和 C++ 主业务揉进同一个服务，业务重启（比如改配置触发的重启）会连带把网络一起断开重新拨号，耗时且不稳定。

**现状：已落地，且比原方案设想的更彻底**：`scripts/cellular_dialup.py` + `systemd/cellular-dialup.service` 是独立的 oneshot 单元（2026-07-09 落地，见 `docs/superpowers/specs/2026-07-09-cellular-dialup-design.md`），跟主程序（`cns-rpi.service`，M7 待建）完全解耦，两者之间**没有** `After=`/`Before=`/`Requires=` 依赖关系。

**明确不采纳的一点**：原方案建议给主程序的 `.service` 加 `After=cns-dial.service`，即主程序要等拨号服务之后才启动。这个反而是不必要的耦合，本项目已经反过来决定：
- `cellular-dialup.service` 拨号失败/重试不应该阻塞主程序启动——5G 只是可选的备用/主用上网通道，不是 MAVLink/UART 链路的依赖。
- 主程序自己的 MQTT 连接判断本来就有"连不上就等下一轮循环再试"的重试逻辑（见 `docs/superpowers/specs/2026-07-09-m5-mqtt-publish-design.md` §3/§7），不需要靠 systemd 的启动顺序去保证网络先就绪。
- 加了 `After=` 反而会让主程序的启动时间被 5G 拨号的重试次数（真机实测过，开机后可能要 2-3 次、每次间隔 15 秒才成功）拖长，没有必要。

M7 设计 `cns-rpi.service` 时，保持跟 `cellular-dialup.service` 之间无依赖关系。

## 5. 硬件防死锁：Systemd 软件看门狗

**背景**：现场最怕"假死"——程序死锁/硬卡死但没崩溃，进程还在，但 MQTT 彻底离线，systemd 的 `Restart=on-failure` 这类基于"进程退出"的重启策略完全捕捉不到这种情况。

**采纳，待落地（M7 范围）**：
- `.service` 文件加 `WatchdogSec=5s`（具体秒数留到 M7 实施阶段跟实际主循环节拍对齐，5s 是原方案给的参考值）。
- C++ 主程序需要周期性调用喂狗。现有 `src/main.cpp` 主循环本来就是"阻塞读 UART（100ms 超时）+ 周期性检查发送心跳"的结构（`kHeartbeatInterval` 那套节拍器），加一次周期性喂狗调用可以直接复用同一个模式，成本很低。
- **实现细节建议**：不建议为了这一个功能去链接 `libsystemd`（引入整个库依赖换一个函数调用不划算）。可以自己实现：往 `$NOTIFY_SOCKET` 环境变量指向的 UNIX datagram socket 发送 `"WATCHDOG=1\n"` 字符串，几行代码就够，systemd 官方文档里 `sd_notify()` 的实现本质就是这几行。

## 6. 部署自动化脚本

**背景**：现场部署不接显示器，安装需要一次性跑完。

**现状：`scripts/install_deps.sh` 已经是更合适的模式，不需要改**：原方案设想"整个脚本用 `sudo ./setup_env.sh` 跑，跑完再 `chown -R pi:pi 项目目录` 把文件归属权还给普通用户"——这个套路是为了解决"用 sudo 跑脚本导致生成的文件变成 root 所有"这个问题，但本仓库现有脚本从设计上就不会产生这个问题：脚本本身以普通用户身份运行，只在真正需要权限的单条命令上单独加 `sudo`（切 apt 源、`apt install`），git 操作和 CMake 编译（2026-07-09 新增的第 5 步）都是以调用者身份跑的，不会产生 root 所有的文件，也就不需要事后 `chown`。

**结论**：本条不采纳"整体套 sudo + 事后 chown"的方案，保持现有 `scripts/install_deps.sh` 的细粒度 sudo 模式。
