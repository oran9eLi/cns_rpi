## 1. 存储防坏卡：OverlayFS 只读根文件系统

**背景**：教学现场设备不能依赖人工值守，学生直接拔电源是常态；SD 卡在断电瞬间的写入很容易导致文件系统损坏、开不了机。

**做法**：
- 开启 OverlayFS，把根文件系统锁死为只读，系统运行期间的写入全部落在内存里的 upper layer，物理断电不会损坏 SD 卡本体。
- 需要写盘的场景（比如远程改配置）不让业务 C++ 程序直接持有 remount 权限：拆一个权限最小化的独立小脚本（例如 `scripts/remount_rw_and_apply_config.sh`），专职做"remount rw → 写配置 → `sync` → remount ro"这几步。
- 业务 C++ 程序通过一条极窄的 `sudoers` 规则（`NOPASSWD: /usr/local/bin/remount_rw_and_apply_config.sh <参数>`，不是给完整 shell 或 `mount` 命令本身的免密权限）去调用这个脚本，业务进程本身不持有 remount 权限。
- 具体触发时机（MQTT 收到什么样的指令、写哪些字段）留给"运行时参数配置化"子项目单独设计，本条只定"只读系统+特权收窄"这个基础设施层面的方案。

**实施设计见 `docs/OverlayFS只读根文件系统设计.md`**（2026-07-20，含现场实测环境数据、配置持久化方案对比、分阶段实施步骤与回退路径）。该文档确认了一个本节未展开的关键约束：配置文件位于根分区，overlay 启用后 helper 的写入会落进内存层、重启即丢，表现为"配置命令返回成功 ACK 但参数回退"的静默失败，因此只读根文件系统与配置持久化必须合并设计、不能分开实施。

## 2. 内存防爆：日志/缓存瘦身

**背景**：OverlayFS 下所有写入都转存到内存（tmpfs），长期运行如果日志/缓存无限增长，内存会被打爆。

**做法**：
- 修改 journald 配置：`Storage=volatile`（只留内存，不落盘）+ `RuntimeMaxUse=16M`（锁死上限，满额后自动滚动覆盖旧日志）。该项已实施：仓库内 `systemd/journald-cns-rpi.conf` 由 `scripts/deploy.sh` 安装到 `/etc/systemd/journald.conf.d/90-cns-rpi.conf`，用 drop-in 而非直接改主配置文件，便于回退。实施时发现 Raspberry Pi OS 自带厂商 drop-in `/usr/lib/systemd/journald.conf.d/40-rpi-volatile-storage.conf` 已把 `Storage` 设为 `volatile`，因此本项的实质作用是补上缺失的 `RuntimeMaxUse` 上限；仍显式重申 `Storage`，避免系统升级改动厂商文件后行为静默变化。
- 业务日志由独立 `logging` 模块统一输出，支持 `debug`/`info`/`warn`/`error` 四级过滤。控制台模式的 stdout/stderr 由 journald 收集；显式配置文件时，所有级别写入同一个文件，按 `logging.max_file_size_kb` 做单文件有界向上滚动，不产生多个历史文件。
- 每条日志使用固定列宽的本地系统时间、进程单调时间和级别字段；系统时间无效时使用 `---------- --:--:--.---` 占位，仍可依单调时间判断事件顺序。
- 任何级别都不逐帧输出 MAVLink 字段或每帧全量 JSON。只有 `debug` 模式在遥测 payload 已成功提交给 MQTT 后，记录同一份紧凑单行 JSON；发布失败只记录 `warn`，不打印 JSON。

## 3. 路径脱敏：禁止相对路径

**背景**：systemd 拉起的进程默认工作目录通常是 `/` 或 `/root`，代码里如果用相对路径打开文件（比如 `open("config.json")`），会因为找不到文件直接闪退。

**做法**：`cns-rpi.service`（主程序的 systemd 单元）的 `ExecStart` 使用可执行文件和配置文件的绝对路径，并显式指定 `WorkingDirectory=/home/dcdw/cns_rpi`，不依赖 systemd 默认工作目录。该项已实施。

## 4. 架构解耦：网络拨号与主程序分离

**背景**：如果把网络拨号和 C++ 主业务揉进同一个服务，业务重启（比如改配置触发的重启）会连带把网络一起断开重新拨号，耗时且不稳定。

**做法**：`scripts/cellular_dialup.py` + `systemd/cellular-dialup.service` 独立成常驻单元，跟主程序（`cns-rpi.service`）之间**不建立** `After=`/`Before=`/`Requires=` 依赖关系。5G 服务是 RM500U AT 口的唯一所有者，负责首次拨号、质量采集和运行期恢复；主程序只读取 `/run/cns-rpi/cellular_status.json`：

- 拨号失败/重试不阻塞主程序启动——5G 只是可选的备用/主用上网通道，不是 MAVLink/UART 链路的依赖。
- 主程序自己的 MQTT 连接判断本来就有"连不上就等下一轮循环再试"的重试逻辑（见 `docs/superpowers/specs/2026-07-09-m5-mqtt-publish-design.md` §3/§7），不需要靠 systemd 的启动顺序去保证网络先就绪。
- 两个服务互不依赖，主程序的启动时间不会被 5G 拨号的重试次数拖长。

## 5. 硬件防死锁：Systemd 软件看门狗

**背景**：现场最怕"假死"——程序死锁/硬卡死但没崩溃，进程还在，但 MQTT 彻底离线，systemd 的 `Restart=on-failure` 这类基于"进程退出"的重启策略捕捉不到这种情况。

**做法**：
- `.service` 文件加 `WatchdogSec=30s`。实施阶段定为 30s：主循环单次迭代上界在百毫秒量级（UART 读 `VMIN=0/VTIME=1` 最多阻塞 100ms，串口发现走后台线程不阻塞主循环），喂狗间隔取超时的 1/3 即 10s，留两次容错，正常运行不会误杀。
- 同时必须加 `NotifyAccess=main`：`Type=simple` 下 `NotifyAccess` 默认为 `none`，systemd 会直接丢弃程序发来的 `WATCHDOG=1`，喂狗形同虚设。
- C++ 主程序周期性调用喂狗。喂狗放在主循环末尾，走到那里说明本轮迭代已完整跑完，没有卡在任何一步。
- 超时时长不在代码里写死，而是从 systemd 注入的 `WATCHDOG_USEC` 读取，unit 文件里的 `WatchdogSec=` 是唯一事实来源，改 unit 不用同步改代码；命令行手工运行时没有该环境变量，整套机制自动降级为空操作。
- 喂狗只证明主循环还在转，不反映业务健康：串口断开和 MQTT 断连各有自己的恢复逻辑且已验证能自愈，不应升级成整个进程重启。
- 不链接 `libsystemd`：往 `$NOTIFY_SOCKET` 环境变量指向的 UNIX datagram socket 发送 `"WATCHDOG=1"` 字符串即可，几行代码，不用为这一个调用引入整个库依赖。实现在 `src/platform/systemd_watchdog.{hpp,cpp}`，纯逻辑部分（喂狗节拍器、环境变量解析）由 `tests/test_systemd_watchdog.cpp` 覆盖。

## 6. 部署自动化脚本

**背景**：现场部署不接显示器，安装需要一次性跑完。

**做法**：`scripts/install_deps.sh` 只在真正需要权限的单条命令上单独加 `sudo`（切 apt 源、`apt install`），git 操作和 CMake 编译以调用者身份跑，不产生 root 所有的文件。依赖安装完成后调用 `scripts/deploy.sh`；后者会先验证普通用户和 sudo 权限，再构建程序，幂等安装配置 helper、`cns-rpi.service` 和 `cellular-dialup.service`，执行 `daemon-reload`，启用并启动两个服务。没有免密 sudo 时会通过 `sudo -v` 正常请求一次密码，不会静默失败。该项已实施。
