版本：2026-07-07
状态：三部分（标准遥测/扩展帧/身份帧）RPi 侧代码均已实现、已在真机验证；身份帧部分的具体字段取值仍有多处待固件侧确认（见第三部分末尾开放问题），不影响 RPi 侧解码逻辑本身。

## 0. 这份文档解决什么问题

固件侧即将开始一次性实现"新 UART 链路发给 RPi"的全部数据发送逻辑，需要一份覆盖全部三类数据（标准遥测、扩展帧、身份帧）的完整对接文档，而不是分散的、只覆盖一部分的文档。

这份文档汇总了 RPi 侧对固件发送格式的全部要求，按 RPi 侧的实现/验证状态分三部分，**确定性依次降低**：

- **第一部分·标准遥测**（对应 RPi 里程碑 M3a）：代码已实现、已在真机验证。RPi 就是调用官方 MAVLink 解码函数把 struct 原样存起来，**没有任何 RPi 特有的格式要求**，固件侧按官方字段定义正常发送即可。
- **第二部分·扩展帧**（对应 RPi 里程碑 M3b）：代码已实现、已在真机验证。RPi 解码器对 `NAMED_VALUE_INT`/`TUNNEL` 的 bit 布局/字节布局有硬编码假设，这部分要求**必须精确**——如果固件侧发送的编码跟本文档给出的字典不完全一致，RPi 会解出错误数据且**不会报错**（M3b 的设计原则是不做语义校验，静默按裸字节解析）。

## 第一部分：标准遥测消息（M3a，已在真机验证）

**核心结论：RPi 按官方 MAVLink 消息定义（`mavlink_msg_*_decode()`）原样解码存储，没有任何 RPi 特有的编码要求，固件侧按官方字段定义正常发送标准消息即可。**

依据：`src/protocol/telemetry_decoder.cpp` 的实现是纯 `msgid` 分发——收到帧后按 `msg.msgid` switch 到对应的官方解码函数（如 `mavlink_msg_heartbeat_decode`），解码结果整个 struct 原样存入 `state_store`，不做任何字段级校验、单位换算或重新解读。因此这部分**不存在"RPi 需要固件侧按特定方式编码"的问题**——只要固件用官方 `mavlink_msg_*_pack()`/`_encode()` 函数正常打包，RPi 就能正确解出。

RPi 目前支持解码的标准遥测消息（详见 `docs/MAVLink消息清单.md` 第 1 节，字段列表不在此重复）：

| 消息 | 说明 |
|---|---|
| `HEARTBEAT` | 心跳，`type`/`autopilot`/`base_mode`/`custom_mode`/`system_status` |
| `GPS_RAW_INT` | GPS 原始定位 |
| `ATTITUDE` | 姿态角/角速度 |
| `GLOBAL_POSITION_INT` | 全球坐标位置 |
| `SYS_STATUS` | 系统状态、传感器 present/enabled/health 位图、电池电压/电流概览 |
| `BATTERY_STATUS` | 电池详细状态 |
| `SCALED_PRESSURE` | 气压/温度 |

注意：`docs/MAVLink消息清单.md` 第 1 节还列了 `STATUSTEXT`，但截至本文档版本，`src/protocol/telemetry_decoder.cpp` 尚未实现该消息的解码分支（该 switch 里没有 `MAVLINK_MSG_ID_STATUSTEXT` 分支）。固件侧仍可以发送 `STATUSTEXT`，只是 RPi 当前会按"未知消息"静默忽略，不落地到 `state_store`，不影响其他消息的解码。

频率：沿用还是重新定，见 `docs/MAVLink消息清单.md` 第 5 节开放问题，本文档不重复讨论。

## 第二部分：扩展帧（M3b，已在真机验证——本文档最重要、要求最精确的部分）

### 2.1 状态与复用要求

`NAMED_VALUE_INT`/`TUNNEL` 这两条消息借用标准 MAVLink 消息的通用载体传递自定义业务数据，字段字典**已从固件源码 `Framework/Src/px4lite_mavlink_tx.c`（`NAMED_VALUE_INT` 的 name/编码）和 `Framework/Inc/px4lite_remote_tunnel.h`（`TUNNEL` payload 布局）取得，不是猜测或反推**，并且已经与 RPi 侧 `src/protocol/extension_decoder.cpp` 的实现逐字段核对一致（`MODSTAT0`/`MODSTAT1` 的 4bit 打包顺序、`BAT2STAT` 的 bit 划分、`MOTORPWM`/`GNSS_SAT` 的字节顺序、两种 `TUNNEL payload_type` 的表头+行字节布局，均已核对，没有偏差）。**这份字典可以直接放心实现。**

**关键要求：这份字典目前是固件"现有 LoRa 链路"（`px4lite_mavlink_tx.c`/`px4lite_remote_tunnel.h`）的实现。固件侧在新 UART 链路上必须复用完全相同的消息类型和字段编码，只是链路和发送频率是新的——不能对字段布局做任何微调（哪怕看起来是"优化"）。** 因为 RPi 侧的解码是硬编码的裸字节解析、不做语义校验，任何编码上的偏差都会导致 RPi 解出错误数据但不报错，这类问题很难在联调时被发现。

### 2.2 `NAMED_VALUE_INT` 字段字典

固件通过 `name`（char[10] 字符串）区分语义，`value` 是 `int32_t`，实际按 `uint32_t` 位布局解读：

| `name`（wire 上实际字符串） | value（int32，按位打包） | 备注 |
|---|---|---|
| `MODSTAT0` | 模块 0-7 的状态，每模块 4 bit：`(state & 0xF) << (i*4)`，`i`=模块序号 0-7 | **每模块 4 bit，模块 0 在最低 4 bit（LSB 端）**，模块 7 在最高 4 bit |
| `MODSTAT1` | 模块 8-13 的状态，编码方式同上，`i`=0 对应模块 8 | 只有 6 个有效模块（8-13），value 的高 2 个 nibble（对应 i=6,7）恒为 0 |
| `BAT2STAT` | bit0-15：电压 mV（`uint16_t`）；bit16-23：电量百分比；bit24：低电压标志（0/1） | bit25-31 未使用 |
| `MOTORPWM` | 最多 4 个电机占空比，每个 1 字节：byte0..byte3 = 电机 0..3 的 `duty_percent` | **byte0 是最低字节（bit0-7），对应电机 0**，逐字节递增对应电机序号递增 |
| `GNSS_SAT` | bit0-7：GPS 可见数；bit8-15：北斗可见数；bit16-23：GPS 使用数；bit24-31：北斗使用数 | 4 个字节各占 1 段，顺序固定为"GPS可见→北斗可见→GPS使用→北斗使用" |
| `ENVHUM` | 相对湿度 × 10（如 535 = 53.5%），取值范围 0-1000 | 整个 `int32` 直接存该整数值，不再拆分 bit 段 |
| `LORASUM` | LoRa 远程查看租约摘要（`active_viewer`/`lease_id`/`remaining_s`） | 主从机之间用的字段，RPi 侧不解码这个 name（未在 `extension_decoder.cpp` 里处理），可以不接入新链路 |

**模块状态枚举 `Px4Lite_State_t`**（`MODSTAT0`/`MODSTAT1` 每个 4bit 字段的取值范围 0-15，目前定义了 0-6）：

| 值 | 含义 |
|---|---|
| 0 | UNINITIALIZED 未初始化 |
| 1 | STARTING 启动/自检中 |
| 2 | ONLINE 在线，主功能可用 |
| 3 | DEGRADED 降级运行 |
| 4 | OFFLINE 超时/失联 |
| 5 | FAILED 故障，需要恢复或人工处理 |
| 6 | DISABLED 配置关闭 |

**模块编号 `Px4Lite_ModuleId_t`**（`MODSTAT0` = 模块 0-7，`MODSTAT1` = 模块 8-13，共 14 个，序号即上表打包顺序里的 `i`）：

| 序号 | 模块 | 序号 | 模块 |
|---|---|---|---|
| 0 | GNSS | 7 | REMOTE_ID |
| 1 | IMU | 8 | DISPLAY |
| 2 | BARO | 9 | CONTROL |
| 3 | BATTERY | 10 | ALARM |
| 4 | LORA | 11 | SYSTEM |
| 5 | 5G | 12 | ESTIMATOR |
| 6 | STORAGE | 13 | BUSINESS |

### 2.3 `TUNNEL` payload 字段字典

固件通过 `payload_type` 字段区分语义，payload 内部字段**全部小端**。

**`payload_type = 0x8001`（`PX4LITE_TUNNEL_PT_ALARM_TABLE`，完整告警表）**：

| 部分 | 字节布局 |
|---|---|
| 表头（2 字节） | `ver`(1 字节) + `active_count`(1 字节) |
| 每行（7 字节） | `source_id`(1) + `fault_code`(2，小端) + `severity`(1) + `active`(1，0/非0) + `age_s`(2，小端) |
| 行数上限 | 最多 14 行（= `PX4LITE_MODULE_COUNT`） |

**`payload_type = 0x8002`（`PX4LITE_TUNNEL_PT_MESSAGE_LOG`，日志增量/心跳）**：

| 部分 | 字节布局 |
|---|---|
| 表头（3 字节） | `latest_seq`(2，小端) + `count`(1 字节) |
| 每条（8 字节） | `sequence`(2，小端) + `message_id`(2，小端) + `time_hhmmss`(3 字节，见下) + `severity`(1) |
| 条数上限 | 最多 9 条（= `PX4LITE_LOCAL_LOG_CAP`）；`count=0` 表示只有心跳（仅 `latest_seq` 有意义，无日志条目） |

`time_hhmmss` 的 3 字节内部顺序：RPi 侧原样按 `[offset+4]`/`[offset+5]`/`[offset+6]` 三个独立字节存储（不作为一个多字节整数解读、不做小端合成），固件侧按"时/分/秒各占 1 字节，顺序为时→分→秒"填充即可，不涉及字节序问题。

RPi 侧解码时对"声明的行数/条数"与"payload 实际长度能容纳的行数/条数"取二者较小值（`std::min`），即如果 `active_count`/`count` 字段的值超过 payload 实际字节数所能容纳的行/条数，RPi 不会越界读，但也不会报错提示——固件侧仍应确保这两个计数字段与实际填充的数据一致，避免 RPi 认为的行数比固件实际填的少（数据被截断）或凭空多算。

**告警严重度枚举 `Px4Lite_AlarmSeverity_t`**（`severity` 字段取值，两个 `TUNNEL` payload 通用）：

| 值 | 含义 |
|---|---|
| 0 | INFO 信息提示 |
| 1 | WARNING 警告，不阻塞主流程 |
| 2 | ERROR 错误，需要记录和恢复 |
| 3 | CRITICAL 严重错误，可能影响关键功能 |
| 4 | FATAL 致命错误，系统应进入失败/保护状态 |

`fault_code`/`message_id` 的具体取值含义由来源模块自行定义，固件源码里没有集中的全局枚举，RPi 侧原样透传、不解读，暂不追加。

### 2.4 帧头字段（适用于全部消息，不只是扩展帧）

MAVLink v2 帧头的 `sysid`/`compid` 是每一帧都必带的固定字段，不依赖具体消息类型，RPi 侧帧解析层在最外层统一提取，不需要在各消息解码器里重复处理。`sysid` 的具体要求见第三部分 3.1（对身份帧的角色号格式化尤其关键，但约束对象是全部消息的帧头，不限于身份帧）。

## 第三部分：身份帧（M3c，代码已实现、已在真机验证——整合自 `docs/固件对接-M3c身份帧字段格式.md`）

RPi 侧对身份数据的原则是"STM32 是权威，RPi 拿到即用，不校验、不重新计算"（`docs/设备标识符.md` §2.3、§4.5），所以本部分除少数结构性硬约束外，大多数取值是**建议/供参考**，最终决定权在固件侧；本部分末尾统一列出需要固件侧决定或确认的开放问题。

### 3.1 帧头 `sysid` 要求（适用于本链路全部消息，不只是身份帧）

**RPi 通过新 UART 收到的每一条 MAVLink 帧，其帧头 `sysid` 必须等于固件的 `PX4LITE_UNIT_ID`。**

依据：`docs/设备标识符.md` §3 —— `DCDW-XXX` 这个编号"来自 `PX4LITE_UNIT_ID`，也是 LoRa/MAVLink 里的 node_id / system_id"。RPi 侧的 `FormatDcdwLabel(sysid)`（M3c 设计）直接读帧头的 `sysid` 字段格式化成 3 位数字补零的 `"DCDW-XXX"`，**不解析任何消息 payload 字段**。如果某条帧的 `sysid` 不等于 `PX4LITE_UNIT_ID`（比如固件转发了别的 system 的帧，或者 RemoteID 心跳用了不同的 sysid），RPi 格式化出来的角色号就会错。

注意区分：这里说的是 **MAVLink 帧头的 `sysid`**（`mavlink_message_t::sysid`，编码/解码函数的 `system_id` 参数），不是下面 `OPEN_DRONE_ID_BASIC_ID` 等消息 payload 里的 `target_system`/`target_component` 字段——那两个字段语义完全不同（表示"这条帧要发给谁"，广播填 0），不要混淆。

### 3.2 `OPEN_DRONE_ID_BASIC_ID`（msgid 12900）逐字段要求

| 字段 | 类型 | 要求 |
|---|---|---|
| `uas_id[20]` | `uint8_t[20]` | 必须填厂商唯一产品识别码整串：`DCDWCNS1` + 12 字符 SN，共 20 字符 ASCII（`docs/设备标识符.md` §2.1）。SN 由 STM32 本地算好（§2.3），RPi **不做校验、直接透传**。 |
| `id_type` | `uint8_t`（`MAV_ODID_ID_TYPE`） | 推荐 `MAV_ODID_ID_TYPE_SERIAL_NUMBER`（=1）。见 3.2.2 警示。 |
| `ua_type` | `uint8_t`（`MAV_ODID_UA_TYPE`） | RPi 只原样存储，不依赖具体取值——由固件侧按产品实际情况选，见 3.2.3。 |

#### 3.2.1 `uas_id` 的 null 填充

`uas_id` 字段官方注释是"UAS ... ID following the format specified by id_type. **Shall be filled with nulls in the unused portion of the field.**"（`mavlink_msg_open_drone_id_basic_id.h`）。

厂商唯一产品识别码正好是 20 字符，理论上不需要 null 填充（20 字节全部用满）。但**如果固件侧的字符串拼装实现产生了不足 20 字符的结果**（比如某个环节出错截断了），未用部分必须显式填 `\0`，不能留脏数据/未初始化内存——RPi 侧用 `strnlen` 提取 `vendor_id`（M3c 设计 `ExtractVendorId`），如果中间夹杂非 null 的垃圾字节，`strnlen` 会在正确长度处截断但后面数据仍留在 buffer 里，不会导致 RPi 崩溃，但如果 20 字节写满且没有 null 终止符，RPi 会按 20 字节全长处理——这正是"厂商识别码恰好 20 字符"这个设计的预期行为，只是要确保这 20 字节**就是**识别码本身，不能是"识别码+垃圾尾巴"拼出来的 20 字节。

#### 3.2.2 `id_type` 警示：ANSI/CTA-2063 vs GB/T 41300

`MAV_ODID_ID_TYPE` 枚举（`common.h`）实际取值：

| 值 | 枚举名 | 官方注释 |
|---|---|---|
| 0 | `MAV_ODID_ID_TYPE_NONE` | No type defined. |
| 1 | `MAV_ODID_ID_TYPE_SERIAL_NUMBER` | Manufacturer Serial Number（**ANSI/CTA-2063 format**）。 |
| 2 | `MAV_ODID_ID_TYPE_CAA_REGISTRATION_ID` | CAA registered ID. Format: [ICAO Country Code].[CAA Assigned ID]. |
| 3 | `MAV_ODID_ID_TYPE_UTM_ASSIGNED_UUID` | UTM assigned UUID (RFC4122). |
| 4 | `MAV_ODID_ID_TYPE_SPECIFIC_SESSION_ID` | 20 字节，用于特定飞行/会话 ID，具体类型由 `uas_id` 首字节指示，由 ICAO 管理。 |

**警示**：`MAV_ODID_ID_TYPE_SERIAL_NUMBER` 的官方注释明确写的是 "ANSI/CTA-2063 format"，但我们实际往 `uas_id` 里填的是 **GB/T 41300 格式**（`DCDWCNS1`+12字符SN，字符集排除 `O`/`I`）的字符串，两者不是同一套格式规范。选 `SERIAL_NUMBER=1` 是"这 5 个选项里语义最接近（都是厂商序列号性质）"，**不是"完全符合标准定义"**。其余 3 个选项（CAA注册号/UTM UUID/特定会话ID）语义都对不上。这个选择需要固件侧确认是否接受，或者认为有更合适的取值——已记入第 4 节开放问题。

#### 3.2.3 `ua_type` 供选列表

`MAV_ODID_UA_TYPE` 枚举取值（`common.h`），**RPi 侧不依赖具体取值，只原样存储**，选哪个是固件侧的产品决定：

| 值 | 枚举名 | 含义 | 值 | 枚举名 | 含义 |
|---|---|---|---|---|---|
| 0 | `NONE` | 未定义 | 8 | `FREE_BALLOON` | 自由气球 |
| 1 | `AEROPLANE` | 固定翼飞机 | 9 | `CAPTIVE_BALLOON` | 系留气球 |
| 2 | `HELICOPTER_OR_MULTIROTOR` | 直升机/多旋翼 | 10 | `AIRSHIP` | 飞艇 |
| 3 | `GYROPLANE` | 旋翼机 | 11 | `FREE_FALL_PARACHUTE` | 自由落体/降落伞（无动力） |
| 4 | `HYBRID_LIFT` | VTOL | 12 | `ROCKET` | 火箭 |
| 5 | `ORNITHOPTER` | 扑翼机 | 13 | `TETHERED_POWERED_AIRCRAFT` | 系留动力飞行器 |
| 6 | `GLIDER` | 滑翔机 | 14 | `GROUND_OBSTACLE` | 地面障碍物 |
| 7 | `KITE` | 风筝 | 15 | `OTHER` | 其他 |

### 3.3 `LOCATION`/`SYSTEM`/`OPERATOR_ID`/`SELF_ID` 字段处理原则

RPi 侧对这 4 个消息的**所有字段原样存储**（对应 `mavlink_open_drone_id_{location,system,operator_id,self_id}_t` 整个 struct 存进 `state_store`），不做单位换算、不做取值校验。

因为实训箱不能飞行，下面逐条列出这 4 个消息里"有明确 unknown 哨兵值"的字段（原文摘自头文件字段注释），**大概率应该填各自的哨兵值，但最终填什么是固件侧的判断，这里只是把可参考的哨兵值准确列出来**。

#### 3.3.1 `OPEN_DRONE_ID_LOCATION`（msgid 12901）

字段级 "If unknown" 数值（摘自 `mavlink_msg_open_drone_id_location.h` 字段注释原文）：

| 字段 | 类型/单位 | If unknown |
|---|---|---|
| `latitude` | int32, degE7 | 0（与 longitude 一起为 0） |
| `longitude` | int32, degE7 | 0（与 latitude 一起为 0） |
| `altitude_barometric` | float, m | -1000 m |
| `altitude_geodetic` | float, m | -1000 m |
| `height` | float, m | -1000 m |
| `timestamp` | float, s | 头文件原文是 "If unknown: 0xFFFF"——注意该字段类型是 `float`，注释按十六进制写的这个值，原样摘录，不做解读/纠正 |
| `direction` | uint16, cdeg | 36100 centi-degrees |
| `speed_horizontal` | uint16, cm/s | 25500 cm/s |
| `speed_vertical` | int16, cm/s | 6300 cm/s |

以下字段本身没有"If unknown"字面注释，但对应到 `common.h` 里的枚举定义，其中都有一个语义上的"未知/未声明"取值（枚举名与字段名前缀一致，为标准 MAVLink 命名习惯，非本头文件逐字段显式标注，供参考）：

| 字段 | 对应枚举 | "未知/未声明"取值 |
|---|---|---|
| `status` | `MAV_ODID_STATUS` | `MAV_ODID_STATUS_UNDECLARED` = 0 |
| `horizontal_accuracy` | `MAV_ODID_HOR_ACC` | `MAV_ODID_HOR_ACC_UNKNOWN` = 0 |
| `vertical_accuracy` | `MAV_ODID_VER_ACC` | `MAV_ODID_VER_ACC_UNKNOWN` = 0 |
| `barometer_accuracy` | `MAV_ODID_VER_ACC` | `MAV_ODID_VER_ACC_UNKNOWN` = 0 |
| `speed_accuracy` | `MAV_ODID_SPEED_ACC` | `MAV_ODID_SPEED_ACC_UNKNOWN` = 0 |
| `timestamp_accuracy` | `MAV_ODID_TIME_ACC` | `MAV_ODID_TIME_ACC_UNKNOWN` = 0 |

`height_reference`（`MAV_ODID_HEIGHT_REF`：0=`OVER_TAKEOFF`，1=`OVER_GROUND`）**没有"未知"选项**，只能二选一，填哪个不影响 RPi 解码，由固件侧决定。

#### 3.3.2 `OPEN_DRONE_ID_SYSTEM`（msgid 12904）

字段级 "If unknown" 数值（摘自 `mavlink_msg_open_drone_id_system.h`）：

| 字段 | 类型/单位 | If unknown / 默认值 |
|---|---|---|
| `operator_latitude` | int32, degE7 | 0（与 longitude 一起为 0） |
| `operator_longitude` | int32, degE7 | 0（与 latitude 一起为 0） |
| `area_ceiling` | float, m | -1000 m |
| `area_floor` | float, m | -1000 m |
| `operator_altitude_geo` | float, m | -1000 m |
| `area_count` | uint16 | 官方注释是"default 1"（不是 unknown 语义，是缺省值） |
| `area_radius` | uint16, m | 官方注释是"default 0"（同上） |
| `timestamp` | uint32, s（Unix 时间戳，起点 2019-01-01 00:00:00） | **头文件没有"If unknown"标注**，具体填 0 还是真实时间戳，记入开放问题 |

枚举类字段的"未声明"取值（同 3.3.1，供参考）：

| 字段 | 对应枚举 | "未声明"取值 |
|---|---|---|
| `classification_type` | `MAV_ODID_CLASSIFICATION_TYPE` | `MAV_ODID_CLASSIFICATION_TYPE_UNDECLARED` = 0 |
| `category_eu` | `MAV_ODID_CATEGORY_EU` | `MAV_ODID_CATEGORY_EU_UNDECLARED` = 0（仅当 `classification_type == EU` 时才有意义，见头文件注释） |
| `class_eu` | `MAV_ODID_CLASS_EU` | `MAV_ODID_CLASS_EU_UNDECLARED` = 0（同上，仅 EU 分类时有意义） |

`operator_location_type`（`MAV_ODID_OPERATOR_LOCATION_TYPE`：0=`TAKEOFF`，1=`LIVE_GNSS`，2=`FIXED`）**没有"未知"选项**，实训箱场景下 `TAKEOFF`（=0，"操作员位置与起飞点相同"）语义上最简单，但具体选哪个由固件侧判断。

#### 3.3.3 `OPEN_DRONE_ID_OPERATOR_ID`（msgid 12905）

| 字段 | 类型 | 说明 |
|---|---|---|
| `operator_id_type` | `uint8_t`（`MAV_ODID_OPERATOR_ID_TYPE`） | 枚举**只有一个取值** `MAV_ODID_OPERATOR_ID_TYPE_CAA`（=0，"CAA 注册的操作者ID"），没有"none/unknown"选项——没有真实 CAA 登记号时怎么处理，记入开放问题 |
| `operator_id[20]` | `char[20]` | 官方注释："Text description or numeric value expressed as ASCII characters. **Shall be filled with nulls in the unused portion of the field.**" 没有真实操作者身份数据时填什么占位值，记入开放问题 |

#### 3.3.4 `OPEN_DRONE_ID_SELF_ID`（msgid 12903）

| 字段 | 类型 | 说明 |
|---|---|---|
| `description_type` | `uint8_t`（`MAV_ODID_DESC_TYPE`） | 0=`TEXT`（自由文本描述飞行目的）/1=`EMERGENCY`（仅 status==EMERGENCY 时用）/2=`EXTENDED_STATUS`（仅 status!=EMERGENCY 时用）。实训箱场景 `TEXT`=0 语义上最通用，但没有"none"选项 |
| `description[23]` | `char[23]` | 官方注释与 `operator_id` 相同："Shall be filled with nulls in the unused portion of the field."。没有真实描述文本时填什么占位值，记入开放问题 |

#### 3.3.5 四个消息共有的 `target_system`/`target_component`/`id_or_mac`

`LOCATION`/`SYSTEM`/`OPERATOR_ID`/`SELF_ID`（以及 `BASIC_ID`）都有这三个字段，RPi 同样原样存储、不做要求：

- `target_system`/`target_component`：官方注释"System/Component ID (0 for broadcast)"，本链路是 STM32→RPi 单点直连，填 0（广播语义）预期是合理默认，但不是 RPi 的强制要求。
- `id_or_mac[20]`：官方注释"Only used for drone ID data received from other UAs"——这是给"转发别的无人机身份数据"场景用的字段，我们自己广播自己的身份时这个字段不适用，预期填全零，但同样不是 RPi 的强制要求（RPi 不校验）。

### 3.4 开放问题（第三部分内部，需固件侧决定或确认）

1. **`id_type` 格式适配**（3.2.2）：`MAV_ODID_ID_TYPE_SERIAL_NUMBER` 官方语义是 ANSI/CTA-2063 格式，我们实际填的是 GB/T 41300 格式字符串，两者不一致。是否接受这个"最接近但不精确匹配"的选择，还是有更合适的取值/处理方式？
2. **`ua_type` 具体取值**（3.2.3）：从列出的 16 个选项里选哪个，是固件侧的产品决定，RPi 不依赖具体取值。
3. **`OPERATOR_ID.operator_id` / `SELF_ID.description` 占位内容**（3.3.3/3.3.4）：V1 阶段大概率没有真实的 CAA 操作者登记号、也没有真实飞行目的描述，这两个字段暂时填什么（全 null？固定占位文本如"教学演示"？）需要固件侧决定。
4. **`OPERATOR_ID.operator_id_type` 没有 unknown 选项**（3.3.3）：枚举只定义了 `CAA`=0 一个值，没有真实 CAA 登记号时是否仍填 0，还是这个字段本身要不要发送，需要固件侧确认。
5. **`SYSTEM.timestamp` 没有文档化的 unknown 哨兵值**（3.3.2）：填 0 还是真实 Unix 时间戳，需要固件侧决定。
6. **accuracy 类字段与枚举的对应关系是推断得出**（3.3.1 表格备注）：`horizontal_accuracy`→`MAV_ODID_HOR_ACC`、`vertical_accuracy`/`barometer_accuracy`→`MAV_ODID_VER_ACC`、`speed_accuracy`→`MAV_ODID_SPEED_ACC`、`timestamp_accuracy`→`MAV_ODID_TIME_ACC` 这几组对应关系是根据字段名与枚举名前缀一致推断的（生成后的 C 头文件本身没有显式标注字段绑定哪个枚举类型，这个信息只存在于原始 MAVLink XML 定义里），建议固件侧对照 `Third_Party/mavlink/common/common.xml`（如果有）或 mavlink.io 官方文档核实一遍。
7. **RemoteID"实名登记标志"字段如何处理**：`docs/设备标识符.md` §6 已经记录这是一个未决问题（不发送 vs 填 NULL），不在 `OPEN_DRONE_ID_*` 消息本身的字段里（是 GB 46750 报文里的另一个数据项），这里一并提醒交叉核对，不重复展开讨论。
8. **STM32 算好的 SN（以及是否连同原始 UID）通过哪个 MAVLink 消息上报给 RPi**：`docs/设备标识符.md` §6 已记录这是开放依赖——SN 已在 `OPEN_DRONE_ID_BASIC_ID.uas_id` 里携带（见 3.2），如果固件侧还需要额外把原始 UID 上报给 RPi 做审计，需要另行确定走哪个消息/字段，本文档未覆盖这部分。

## 第四部分：开放问题汇总

按所属部分归类，方便固件侧一次性看完所有待确认事项：

**第一部分·标准遥测**：无待确认的格式类开放问题（无 RPi 特有编码要求）；频率/是否需要 `STATUSTEXT` 等消息范围调整，见 `docs/MAVLink消息清单.md` 第 5 节。

**第二部分·扩展帧**：字段字典已核对一致，无格式类开放问题；`fault_code`/`message_id` 的具体业务含义暂无集中枚举，不影响 RPi 按字节布局解码/透传；频率沿用/重新定，见 `docs/MAVLink消息清单.md` 第 5 节。

**第三部分·身份帧**（详见 3.4）：

1. `id_type` 是否接受 "SERIAL_NUMBER（ANSI/CTA-2063 语义）借用于 GB/T 41300 格式字符串" 这一不精确匹配。
2. `ua_type` 具体取值（固件侧产品决定，不阻塞 RPi）。
3. `OPERATOR_ID.operator_id` / `SELF_ID.description` 无真实数据时的占位内容。
4. `OPERATOR_ID.operator_id_type` 无 CAA 登记号时是否仍填 0，或该字段是否发送。
5. `SYSTEM.timestamp` 填 0 还是真实 Unix 时间戳。
6. accuracy 类字段与对应枚举的绑定关系建议对照官方 XML/文档核实。
7. RemoteID"实名登记标志"字段处理方式（交叉见 `docs/设备标识符.md` §6）。
8. STM32 算好的 SN／原始 UID 上报给 RPi 的具体消息通道确认（SN 本身已走 `uas_id`，原始 UID 若需额外上报待定）。

此外，`docs/MAVLink消息清单.md` 第 5 节记录了跨越第一、二、三部分共通的开放问题（频率重定、`OPEN_DRONE_ID_*` 接入新链路需要固件侧新写调度表、两条 `HEARTBEAT` 是否合并），不在此重复列出，请一并查阅。
