# CNS 实训箱 · RPi 端 V1 设计文档

版本：2026-07-01（修订：加入命令下行链路与可扩展性设计；系统实测为 trixie，语言改 C++23）
状态：设计修订中，待确认后定稿
适用范围：Raspberry Pi 5（trixie / Debian 13, ARM64）端数据汇聚与回传程序

## 1. 背景与项目定位

CNS（通信/导航/监视）实训箱是面向低空经济方向职教教学的产品，由多端系统组成：

- **STM32F407 + FreeRTOS**：实训箱主控固件（固件仓库：`oran9eLi/Formal_Framework`，分支 `fj-lora`），负责外设控制、LoRa 点对点互联、RemoteID 身份广播等，由固件架构负责人独立维护，本设计不涉及固件内部实现。
- **树莓派（Raspberry Pi 5）**：数据汇聚与回传节点，本设计的范围。
- **两台下游服务器**：
  - **硬件部服务器**：RPi 直接发布/订阅 MQTT 数据的目的地。V1 阶段该服务器尚未就绪，暂用开发机（Ubuntu）上的 Mosquitto 实例代替。
  - **软件部服务器**：接收硬件部服务器桥接过来的遥测数据，也是 V1 已知的控制命令来源。硬件部服务器到管控中心的 MQTT 桥接不在本设计范围内。

RPi 的职责是**遥测/身份数据的统一汇总入口，同时也是控制命令的下行转发节点**：

- **上行**：从 STM32 收 MAVLink 帧，解析为内部状态，重组为 JSON，作为 MQTT client 发布到硬件部服务器。
- **下行**：接收外部控制命令（V1 已知来源是管控中心，通过 MQTT 下发；不保证是唯一来源），转发给 STM32 执行。

V1 没有本地显示屏。V2 会引入 Qt 显示，**取代**（不是新增旁路）当前 STM32 侧的并口 LCD 显示，同时接入摄像头 + OpenCV 视觉处理。V2 不在本设计范围内，但本设计的内部结构按可扩展性要求设计，让 V2 的这些新增消费者/来源能够接入而不用改动 STM32 对接部分。

## 2. 数据流架构

```
STM32F407
  │ 新分配的 UART（口号/波特率待固件侧确定，需支持双向收发）
  │ MAVLink v2, common dialect, 双向：
  │   上行(STM32→RPi): HEARTBEAT / 遥测 / 扩展帧 / OPEN_DRONE_ID_*
  │   下行(RPi→STM32): COMMAND_LONG，STM32 回 COMMAND_ACK
  ▼▲
Raspberry Pi 5
  上行：UART收帧 → 解码 → 内部状态 → JSON → MQTT publish
  下行：MQTT subscribe(命令topic) → 内部命令分发 → 编码COMMAND_LONG → UART发送
        → 收到COMMAND_ACK → 状态回传
  ▼▲
硬件部服务器（V1: 开发机 Mosquitto 代替）
  │ MQTT bridge（服务器间，不在本设计范围）
  ▼▲
软件部服务器（管控中心，V1 已知命令来源）→ 软件部前端
```

**关键澄清（对应此前误解的修正）**：

- RPi 与 STM32 之间目前**没有已分配的 UART**，需要固件侧新开一个（如 UART5），且需要同时支持 TX 和 RX（此前只确认了发送侧，命令下行需要固件侧的 RX 中断/DMA 也配置好）。这是 M2 里程碑的外部阻塞项。
- RemoteID（OpenDroneID）数据当前的实际去向是 `STM32 → UART4 → ESP32-S3 → BLE/Wi-Fi 广播`，**不经过 RPi**。RPi 需要的是一条新链路，把 LoRa 侧遥测数据和 RemoteID 身份数据**都**发一份过来，而不是复用 UART4。
- RPi **不需要在本地跑 Mosquitto**，也不需要 bridge 配置——RPi 就是普通 MQTT client，直连硬件部服务器发布/订阅；"桥接"发生在硬件部服务器与管控中心之间，与 RPi 无关。
- **启动时序依赖（为 V2 设备间通信预留）**：MQTT topic 里的设备寻址字段用厂商唯一产品识别码（详见第 9 节），这个值要等 STM32 上报 `OPEN_DRONE_ID_BASIC_ID` 帧解码出来才能拿到，所以 RPi 的启动顺序是"先建 UART 链路 → 收到身份帧、解出唯一 ID（'身份就绪'）→ 才初始化 MQTT client 并连接/订阅"，不是"读完配置就直接连 MQTT"。

## 3. 架构原则：解耦与可扩展性

V1 只有一个上行消费者（MQTT 发布）和一个已知下行来源（管控中心），但 V2 会加 Qt 本地显示（取代 STM32 侧显示，需要读取和 MQTT 一样的解码状态）和摄像头/OpenCV（独立的新增数据源，可能还要跟遥测数据关联）。为了不在 V2 时重写 STM32 对接部分，V1 的内部结构按下面两条原则设计：

1. **上行：解码与消费者分离。** UART/MAVLink 解码层只负责"把 STM32 发来的数据变成一份内部状态"，不关心谁来读。V1 只有 MQTT 发布这一个消费者读这份状态；V2 加 Qt 渲染、CV 管线时，只是多接一个消费者读同一份状态，解码层不用动。
2. **下行：命令来源与执行转发分离。** RPi 向 STM32 转发命令的逻辑（编码 COMMAND_LONG、发送、等 ACK）只认一个内部命令队列，不关心命令从哪来。V1 只接入 MQTT 一个来源；如果以后出现其他来源（比如 V2 本地 Qt UI 上的手动操作按钮），只是多接一个来源塞命令进同一个队列，转发逻辑不用动。

这两条原则直接体现在第 8 节目录结构里的 `state/` 和 `command/` 模块划分上。

## 4. MAVLink 协议层（上行）

固件侧全部使用**官方 MAVLink common dialect**（`Third_Party/mavlink/common/`），没有自定义 dialect XML：

- 标准消息通过官方编码函数直接发送（`HEARTBEAT`、`GPS_RAW_INT`、`ATTITUDE`、`GLOBAL_POSITION_INT`、`SYS_STATUS`、`BATTERY_STATUS`、`SCALED_PRESSURE` 等）。
- 自定义业务数据借用标准消息的通用载体，不新增 message id，`NAMED_VALUE_INT` 靠 `name` 字段、`TUNNEL` 靠 `payload_type` 字段区分语义。
- 身份/RemoteID 消息：`OPEN_DRONE_ID_BASIC_ID` / `LOCATION` / `SYSTEM` / `OPERATOR_ID` / `SELF_ID`。

**RPi V1 需要对接的上行消息范围**：`HEARTBEAT` + 上述遥测类 + `NAMED_VALUE_INT`/`TUNNEL` 扩展帧 + `OPEN_DRONE_ID_*` 身份帧。

**实现依据**：RPi 项目直接 vendor 固件仓库 `Third_Party/mavlink/{common,standard,minimal}` 下的官方生成头文件（同一份文件，两端消息定义强一致，避免字段错位）。

### 4.1 `NAMED_VALUE_INT` 字段字典（已从固件源码 `Framework/Src/px4lite_mavlink_tx.c` 取得，非猜测）

**2026-07-07 跟固件 `Formal_Framework` PR#1 同步后订正**：固件已把 `ENVHUM`→`HUMIDITY`、单条 `MOTORPWM`→`MOTOR12`/`MOTOR34` 双帧，并新增 2 条 RPi 专属遥测（`LORASTAT`/`RIDSTAT`，只发 USART1，不上 LoRa）。下表已按最新固件源码更新，这是 M3b 字段字典的订正，不是新里程碑。

**2026-07-08 跟固件侧对齐后订正**：`BAT2STAT` 已废弃——电池2改成跟电池1一样，走官方 `BATTERY_STATUS` 消息（用 `id=1` 区分电池1的 `id=0`），不再借用 `NAMED_VALUE_INT` 自定义字段。RPi 侧 `Battery2Status`/`battery2_status`/`UpdateBattery2Status`/`BAT2STAT` 解码分支已在 M4 落地时一并删除，详见 `docs/superpowers/specs/2026-07-07-m4-json-payload-design.md`。

| `name`（wire 上实际字符串） | value（int32）编码 |
|---|---|
| `MODSTAT0` | 模块 0-7 的状态，每模块 4 bit，LSB 在前：`(state & 0xF) << (i*4)` |
| `MODSTAT1` | 模块 8-13 的状态，编码同上（只有 6 个有效模块，高 2 个 nibble 恒为 0） |
| `MOTOR12` | `[0:7]`电机1占空% `[8:15]`电机2占空% `[16]`run_state(0/1) `[24:31]`speed_level% |
| `MOTOR34` | `[0:7]`电机3占空% `[8:15]`电机4占空% `[16]`run_state `[24:31]`speed_level（`run_state`/`speed_level` 两帧携带同一份整机状态的冗余拷贝） |
| `GNSS_SAT` | bit0-7：GPS 可见数；bit8-15：北斗可见数；bit16-23：GPS 使用数；bit24-31：北斗使用数 |
| `HUMIDITY` | 相对湿度 × 10（如 535 = 53.5%），取值 0-1000（此前叫 `ENVHUM`，已按固件实际改名） |
| `LORASTAT` | RPi 专属，只发 USART1：`[0:15]`丢包率×10 `[16:23]`LoRa节点ID `[24]`LoRa模块在位标志 `[25:27]`LoRa链路状态枚举(`Px4Lite_State_t`) |
| `RIDSTAT` | RPi 专属，只发 USART1：`[0:15]`位置广播成功计数低16位(增量) `[16:31]`编码/提交错误计数低16位(增量)，`time_boot_ms`=RemoteID最近一次成功提交时间 |
| `LORASUM` | LoRa 远程查看租约摘要（active_viewer/lease_id/remaining_s），主从机之间用，跟 RPi 关系不大，可以不接 |

**模块状态枚举 `Px4Lite_State_t`**（`MODSTAT0`/`MODSTAT1` 每个 4bit 字段的取值）：

| 值 | 含义 |
|---|---|
| 0 | UNINITIALIZED 未初始化 |
| 1 | STARTING 启动/自检中 |
| 2 | ONLINE 在线，主功能可用 |
| 3 | DEGRADED 降级运行 |
| 4 | OFFLINE 超时/失联 |
| 5 | FAILED 故障，需要恢复或人工处理 |
| 6 | DISABLED 配置关闭 |

**模块编号 `Px4Lite_ModuleId_t`**（`MODSTAT0`=模块0-7，`MODSTAT1`=模块8-13，共 14 个）：
`0=GNSS 1=IMU 2=BARO 3=BATTERY 4=LORA 5=5G 6=STORAGE 7=REMOTE_ID 8=DISPLAY 9=CONTROL 10=ALARM 11=SYSTEM 12=ESTIMATOR 13=BUSINESS`

### 4.2 `TUNNEL` payload 字段字典（来自 `Framework/Inc/px4lite_remote_tunnel.h`）

**`payload_type = 0x8001`（`PX4LITE_TUNNEL_PT_ALARM_TABLE`，完整告警表）**，小端：

- 表头 2 字节：`ver`(1) + `active_count`(1)
- 每行 7 字节：`source_id`(1) + `fault_code`(2) + `severity`(1) + `active`(1) + `age_s`(2)
- 最多 14 行（= `PX4LITE_MODULE_COUNT`）

**`payload_type = 0x8002`（`PX4LITE_TUNNEL_PT_MESSAGE_LOG`，日志增量/心跳）**，小端：

- 表头 3 字节：`latest_seq`(2) + `count`(1)
- 每条 8 字节：`sequence`(2) + `message_id`(2) + `time_hhmmss`(3) + `severity`(1)
- 最多 9 条（= `PX4LITE_LOCAL_LOG_CAP`），`count=0` 表示只有心跳（仅 `latest_seq` 有意义）

**告警严重度枚举 `Px4Lite_AlarmSeverity_t`**（`severity` 字段取值，两个 TUNNEL payload 通用）：

| 值 | 含义 |
|---|---|
| 0 | INFO 信息提示 |
| 1 | WARNING 警告，不阻塞主流程 |
| 2 | ERROR 错误，需要记录和恢复 |
| 3 | CRITICAL 严重错误，可能影响关键功能 |
| 4 | FATAL 致命错误，系统应进入失败/保护状态 |

`fault_code`/`message_id` 的具体取值含义由来源模块自行定义，固件源码里没有集中的全局枚举，暂不追加。

MAVLink v2 帧头（`sysid`/`compid`）是每一帧都必带的固定字段，不依赖具体消息类型，因此帧解析层可以在最外层统一提取，不需要在各消息解码器里重复处理。

## 5. 命令下行链路

- **命令来源**：V1 确认的来源是管控中心，通过 MQTT 下发 JSON 命令。**不保证是唯一来源**，因此命令接收这一侧按接口抽象设计（见第 3 节、第 8 节 `command/` 模块），后续接入其他来源时只需新增一个实现，不改动转发/编码逻辑。
- **下行流程**：外部来源（JSON）→ 内部命令队列/分发器 → 编码为 MAVLink `COMMAND_LONG` → UART 发送给 STM32 → 等待 `COMMAND_ACK` → 结果状态（成功/失败/超时）回传（V1：发布到 MQTT 的 ack/状态 topic）。
- **鉴权边界**：固件侧现有原则是"所有外部命令统一进入 Command 执行器，由 STM32 校验权限、目标、状态和安全条件"，即 STM32 是最终把关方，不假设 RPi 转发的命令是可信的。RPi 侧至少要做 JSON schema 校验（字段类型、command id 是否在已知范围内），避免把格式错误的数据编码成 MAVLink 帧发出去，但不需要重复实现 STM32 那一层的业务安全校验。
- **待确定项**：具体 `COMMAND_LONG` 命令集合（command id、7 个 float 参数的含义映射）需要固件侧 `px4lite_command.c` 提供映射表，这是新的开放依赖项（见第 11 节）。

## 6. 身份策略

详见 `docs/设备标识符.md`。核心结论：`DCDW-XXX` 是学校对内编号（兼固件内部角色标识），只在单个学校范围内唯一，不能跨校当全局键用；权威设备键是厂商唯一产品识别码（参照 GB/T 41300 结构，SN 段由 STM32 芯片 UID 哈希派生），V1 阶段先用 RPi 硬件序列号过渡。此前设想的"服务器端加盐去重注册服务"已作废，不再需要。

## 7. 技术选型

| 模块 | 选型 | 理由 |
|---|---|---|
| 语言 | C++23 | 实机确认 RPi 跑的是 trixie（Debian 13），自带 GCC 14，`std::expected` 等 C++23 特性支持完整、零风险；`std::span`（C++20 起）对 `uart/`、`protocol/` 里裸字节缓冲区的处理是实打实的收益，`std::expected` 对 `uart/`/`protocol/`/`command/`/`config/` 里多处"操作可能失败"的场景（帧解码、命令校验、配置加载）是标准化的收益，比每个模块各自手搓 `Result<T,E>` 更省事。（此前误以为系统是 Bookworm/GCC 12 才定的 C++20，已修正） |
| 构建 | CMake，RPi 本地原生编译 | RPi 5 算力足够，省去交叉编译工具链维护成本 |
| 串口 | POSIX termios | 帧同步交给官方 MAVLink 库的 `mavlink_frame_char_buffer()`，不自造协议解析；同一封装支持双向收发 |
| MAVLink | 官方 `mavlink/c_library_v2`（vendor 头文件，来自固件仓库同一份） | 两端协议定义强一致，避免版本漂移；编码/解码同一套库双向都能用 |
| MQTT | libmosquitto（C API），纯 client，无本地 broker | apt 直接可用；既发布遥测也订阅命令，与硬件部服务器上的 Mosquitto 是同一生态 |
| JSON | nlohmann/json（header-only） | 内部状态 → JSON 序列化，命令 JSON 反序列化，及本地配置文件读取 |
| 日志 | 独立 `logging` 模块 | 四级过滤，支持 journald 控制台模式与单文件有界滚动，便于教学现场排障和事后追溯 |
| 进程管理 | systemd service，开机自启 + 崩溃自动重启 | 教学现场设备不能依赖人工值守 |
| 配置 | JSON 配置文件 | 串口设备路径/波特率、MQTT broker 地址、topic、设备标签等现场可调，不硬编码 |

## 9. MQTT 收发设计

- RPi 是 MQTT client，不在本机跑 Mosquitto broker。RPi **既发布遥测，也订阅命令**。
- **broker 地址目前有两套，当前现场采用内网**：
  - 内网直连 `192.168.10.211:18884` —— **当前 `dcdw@192.168.11.4` 现场设备使用的就是这套**，用于实验室联调阶段。
  - 公网入口 `112.124.52.232:1883` —— 流量经内网穿透转发到硬件部服务器 `192.168.11.3:1883`，为正式部署预留，尚未切换。
  切换时机与 §9.2 的 MQTT TLS/认证/ACL 一并推进：在公网上跑匿名明文 MQTT 不是可接受的生产状态，因此「切公网」和「上 TLS + 每设备凭据」应当是同一次变更，不要先切公网再补安全。地址只改 `config/config.json`，代码不写死。
- 设备 topic 统一采用 `{namespace}/{vendor_id}/{suffix}`。遥测为 `{namespace}/{vendor_id}/telemetry`，注册为 `{namespace}/{vendor_id}/registration`；服务端订阅 `{namespace}/+/registration` 即可发现全部设备，无需预知 `vendor_id`。
- telemetry 使用 QoS 0、`retain=false`。遥测是按节拍刷新的实时值，不是设备状态的权威存档：retained 遥测会让新订阅者立刻收到最后一帧，却分不清那是实时数据还是设备掉电前的陈旧快照。设备是否在线由 registration topic 的 retained `online`/`offline` 表达，不由遥测兼任。QoS 取 0 是因为 1Hz 固定节奏发布下丢帧可由下一拍自然补上，不值得为此付出 QoS 1/2 的确认开销和积压风险。QoS 仍保留为 `mqtt.topics.telemetry.qos` 可配置项，默认 0；`retain` 不可配置，固定为 `false`。
- registration 使用 QoS 2、retained。MQTT 首次连接、自动重连或角色号变化时发布 `online`；异常断线由 Last Will 发布 `offline`，`SIGINT`/`SIGTERM` 正常退出时主动发布 `offline`。
- 服务端以 `vendor_id` 为主键幂等 upsert 注册信息。`dcdw_label` 未就绪时不阻塞连接、注册消息省略该字段；后续就绪或变化时重发 registration。设备表元数据只从 registration 更新，不从 telemetry 补字段；offline 缺失的元数据字段不清空数据库旧值。
- 配置中的 `mqtt.connection.client_id` 是前缀，实际 MQTT Client ID 为 `{前缀}-{vendor_id}`，避免多台设备使用同名 Client ID 互相踢线。
- **Topic 里的设备寻址字段必须用厂商唯一产品识别码（`docs/设备标识符.md` 里的权威全局键），不能用 V1 过渡期的 RPi 硬件序列号**——否则 V2 切换到权威身份来源时 topic 名字会变，所有订阅方（管控中心、以后可能出现的其他设备）都要跟着重新订阅。连带影响：topic 命名方案的最终定稿要等 M3c（身份帧解码，能从 `OPEN_DRONE_ID_BASIC_ID` 解出这个 ID）完成之后才能真正拍板，不是到 M5 才第一次考虑；不需要因此调整现有里程碑顺序（M3c 本来就排在 M5 之前），但 M5 实现时要显式处理第 2 节提到的"等身份就绪再连 MQTT"这个启动依赖。
- **V2 设备间通信的预留**：本机向别的设备下命令（往对方的命令 topic 发布），不需要新增模块——直接复用本节这套 `mqtt/mqtt_client.hpp` 的 publish 能力即可，不经过本机 `command/command_dispatcher`（那套是"RPi → 自己的 STM32"专用的转发链路，往别的设备发命令不需要走这条链路）。反向"读别的设备状态"的预留位见第 8 节 `remote_state/`。
- Broker 地址、端口、认证信息（V1 本地调试阶段可明文，后续预留 TLS + 用户名密码字段位）均放配置文件，不写死。
