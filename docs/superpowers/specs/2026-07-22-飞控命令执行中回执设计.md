# 飞控命令执行中回执设计

日期：2026-07-22

## 1. 背景

当前树莓派在收到飞控命令后，将命令编码为 MAVLink `COMMAND_LONG`
并发送给 STM32，但不会在发送成功后立即上报状态。服务器只有在收到
STM32 的 `COMMAND_ACK` 或等待超时后，才能收到该命令的回执。

本次调整要求树莓派在确认飞控命令已成功写入 STM32 串口后，立即上报
一次“执行中”回执；收到固件回复后，再按现有规则上报最终回执。

## 2. 范围

本次仅调整飞控命令链路：

- 适用于 `set_motor_pwm`、`takeoff`、`land` 和 `emergency_stop`。
- 树莓派侧不要求新增固件协议；固件将最终 `COMMAND_ACK` 的发送时机调整为
  动作完全结束后，MAVLink 命令编号和参数映射保持不变。
- 不修改配置修改命令的处理状态、回执和重启流程。
- 不新增 `completed` 状态；继续使用 `accepted` 表示固件已回复成功，
  业务上将其解释为“完成”。

## 3. 命令时序

### 3.1 正常时序

1. MQTT 客户端收到本设备 `control/set` topic 上的 JSON 命令。
2. 树莓派完成 JSON 校验、命令去重和当前事务状态检查。
3. 树莓派将命令编码为 `COMMAND_LONG` 并写入 STM32 串口。
4. 串口写入成功后，树莓派立即发布非最终的 `in_progress` 回执。
5. 树莓派继续等待与当前命令匹配的 STM32 `COMMAND_ACK`。
6. 动作完全结束后，固件发送最终 `COMMAND_ACK`；树莓派按现有 MAVLink
   结果映射发布最终回执。

立即回执的 JSON 格式为：

```json
{
  "command_id": "flight-001",
  "command": "takeoff",
  "mavlink_command": 31091,
  "status": "in_progress",
  "result_code": "pending",
  "message": "命令已下发，正在等待单片机应答"
}
```

固件在动作成功结束后返回 `MAV_RESULT_ACCEPTED`，最终回执继续使用：

```json
{
  "command_id": "flight-001",
  "command": "takeoff",
  "mavlink_command": 31091,
  "status": "accepted",
  "result": 0,
  "result_code": "accepted",
  "message": "单片机已执行命令"
}
```

### 3.2 固件中间进度

如果固件返回 `MAV_RESULT_IN_PROGRESS`，树莓派继续按现有逻辑上报包含
`progress` 和 `result_param2` 的 `in_progress` 回执，并延长该命令的等待期限。
树莓派主动上报的 `pending` 和固件上报的实际进度可以先后出现，服务器应将
它们视为同一 `command_id` 事务的中间状态。

树莓派主动发布的 `pending` 是每条命令的基础执行中状态，因此固件无需在
动作刚开始时重复发送 `MAV_RESULT_IN_PROGRESS`。如果固件需要报告实际进度，
仍可选择发送该结果；无论是否提供中间进度，最终 ACK 都应在动作完全结束后
发送。

## 4. 异常和幂等处理

- 命令格式非法、命令不支持、STM32 串口不可用、STM32 身份未学习或串口
  写入失败时，不得发布 `in_progress`，继续返回现有 `rejected` 回执。
- 等待 STM32 回复超时时，发布现有 `timeout` 最终回执。
- 固件返回拒绝、不支持、执行失败或取消时，发布现有 `rejected` 最终回执，
  并保留具体 `result_code`。
- 相同 `command_id` 和相同命令在执行期间重复到达时，不重复下发
  STM32，返回当前 `in_progress` 状态。
- 已完成命令重复到达时，继续重放缓存的最终回执。
- 最终回执保持现有有界幂等缓存和超时后迟到 ACK 隔离逻辑。

## 5. MQTT 发布策略

立即回执复用现有 `control/ack` topic 和 QoS 配置，不新增 topic 或配置项。
树莓派在串口发送成功后立即调用 MQTT 发布，不等待 QoS 2 完整交付确认，
避免阻塞主循环中的串口接收。

命令由当前 MQTT 连接接收，紧接着的中间回执通常能够加入同一连接的
发布队列。如果此次发布未能加入 MQTT 队列，应记录告警，但不撤销已经
成功写入 STM32 的命令，也不将其改判为执行失败。最终回执仍按现有机制
处理。

## 6. 代码边界

- `control_command` 模块继续负责构造标准化的 `in_progress` 回执，避免
  `main.cpp` 重复组装 JSON。
- 主程序只在 `SerialLink::SendMessage()` 返回成功后发布该回执。
- `ControlTransaction` 的 pending、最终回执、幂等缓存和超时处理语义保持不变。
- 配置命令模块不作任何修改。

## 7. 测试与验收

自动化测试覆盖：

1. 标准执行中回执包含正确的 `command_id`、`command`、`mavlink_command`、
   `status=in_progress` 和 `result_code=pending`。
2. 首次命令仍只允许一次 STM32 串口下发。
3. 执行中重复命令不重复下发，仍返回 `in_progress`。
4. STM32 最终 ACK 仍正确映射为 `accepted` 或 `rejected`。
5. 串口不可用、身份未学习和发送失败时不生成成功下发的中间回执。
6. 超时、迟到 ACK 隔离和已完成回执重放行为保持不变。

集成验收时，对同一 `command_id` 应先观察到 `in_progress/pending`，随后
观察到固件 ACK 对应的最终状态。
