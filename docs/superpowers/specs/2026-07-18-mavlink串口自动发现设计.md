# MAVLink 串口自动发现设计

**日期：** 2026-07-18
**状态：** 已实施并完成树莓派实机验收

## 1. 背景与目标

STM32 UART6 通过 CH340 转为 USB 串口连接 Raspberry Pi。Linux 分配的
`/dev/ttyUSBn` 会随设备插拔和枚举顺序变化；当前把该路径写死在
`config.json` 中，会导致主程序在编号变化后打开串口失败并退出。固件显示屏只能看到
5G 离线，无法区分 5G 故障和树莓派未找到 STM32 串口。

本次改动的目标是：主程序不依赖 CH340 的 VID/PID、序列号或动态设备号，而是通过
收到 CRC 正确的 MAVLink 帧确认 STM32 链路；启动时找不到串口就持续等待，运行中断线
后也自动重新发现。整个过程必须输出明确且限频的中文日志。

## 2. 已确认的现场事实

- 物理链路为 `STM32 UART6 -> CH340 -> Raspberry Pi USB`。
- 当前 CH340 的 USB VID/PID 为 `1a86:7523`，但不同批次或替代芯片不能保证相同，
  因此 VID/PID 不能作为正确性的必要条件。
- RM500U-CNV 会同时枚举多个 `/dev/ttyUSB*` AT/诊断端口。
- 除 STM32 固件外，现场没有其他设备会向树莓派串口发送 MAVLink 协议。
- 固件会按固定频率持续发送 HEARTBEAT 和数据帧，不因收到树莓派回复而停止心跳。
- 固件在 USART6 上发送的合法 MAVLink 帧使用同一设备身份；HEARTBEAT 的 `sysid` 为
  UID 派生值、`compid` 为 193。树莓派从 HEARTBEAT 学习这些值，不能写死 `sysid`。
- 正常 MAVLink 解析继续使用仓库内的官方 MAVLink 解析器和 CRC 校验，不自造帧判据。

## 3. 方案选择

采用“顺序探测候选串口，收到任意 CRC 正确的完整 MAVLink 帧即确认”的方案。

不采用以下方案：

- 只按 `1a86:7523` 或 `/dev/serial/by-id` 匹配：不能兼容替代 USB 转串口芯片，且当前
  CH340 没有真正唯一的硬件序列号。
- 只等待 `type=18、compid=193` 的 HEARTBEAT：虽然当前固件会持续发心跳，但串口发现
  只需要回答“哪个端口承载 MAVLink”，接受任意合法数据帧能更快完成发现，并避免把
  端口发现和具体业务消息类型耦合。
- 并行占用所有候选端口：会同时占用 RM500U 的多个端口，复杂度和资源冲突风险没有
  必要。

VID/PID、USB interface number 和稳定符号链接可以用于诊断或候选排序，但不得成为
确认 STM32 链路的必要条件。链路身份最终只由有效 MAVLink 帧确认。

## 4. 配置语义

`serial.device` 支持两种模式：

```json
{
  "serial": {
    "device": "auto",
    "baud": 115200
  }
}
```

- `"auto"`：正式部署模式，持续自动发现 MAVLink 串口。
- 明确设备路径：开发与故障诊断模式，只尝试指定设备，不扫描其他端口；打开失败或
  没有收到有效 MAVLink 帧时仍持续重试，不退出进程。

`config/config.example.json` 默认使用 `"auto"`。部署时把树莓派实际配置迁移为
`"auto"`。程序不得把本次发现的 `/dev/ttyUSBn` 写回配置，因为该编号下次仍可能变化。

## 5. 组件边界

新增独立的串口发现模块 `uart/mavlink_port_discovery`，避免把设备枚举、探测时序和
日志限频继续堆入 `main.cpp`。该模块负责：

1. 根据配置生成候选设备列表；
2. 顺序打开候选串口；
3. 在单个候选的有限时间窗口内读取字节；
4. 用官方 MAVLink 解析器校验完整帧和 CRC；
5. 成功时返回已打开的 `MavlinkLink`、实际设备路径和探测期间收到的第一帧；
6. 一轮失败后释放所有候选端口并等待下一轮。

发现模块不负责 MQTT、遥测状态、STM32端点学习、5G状态心跳或控制命令业务。这些
行为仍由现有业务模块负责。发现结果携带探测到的首帧，业务层照常处理：任意合法帧
都可确认物理端口，但只有符合固件约定的 HEARTBEAT 用于学习 STM32 `sysid/compid`。

`MavlinkLink`/底层串口接口需要把以下结果明确区分：

- 当前没有完整帧；
- 收到一条完整帧；
- 串口发生不可恢复的读取或设备错误。

不能再把设备断开与“暂时没有数据”都表示成同一个空结果。

业务层同时维护最后一次收到合法 MAVLink 帧的单调时钟时间。发现成功时用首帧时间
初始化，后续只有通过 CRC 校验的完整 MAVLink 帧才能刷新；普通字节、AT响应、CRC
错误帧和空读取均不能刷新。

## 6. 启动发现流程

主程序启动流程为：

```text
读取配置并初始化日志
        -> 持续发现 MAVLink 串口
        -> 找到有效帧并确认端口
        -> 处理探测期间收到的第一帧
        -> 进入现有遥测、MQTT和控制命令主循环
```

自动模式枚举 `/dev/ttyUSB*` 和 `/dev/ttyACM*`，排序、去重后逐个探测。RM500U 的
多个 USB 串口通常一次枚举并占据连续编号，独立 CH340 往往位于连续区间的一端，
因此候选采用“两端交替”顺序以缩短常见场景的发现时间。例如：

```text
数值自然排序：ttyUSB0, ttyUSB1, ttyUSB2, ttyUSB3, ttyUSB4, ttyUSB5, ttyUSB6
实际探测顺序：ttyUSB0, ttyUSB6, ttyUSB1, ttyUSB5, ttyUSB2, ttyUSB4, ttyUSB3
```

设备尾号必须按整数排序，不能使用会把 `ttyUSB10` 排在 `ttyUSB2` 前面的字符串字典
序。`ttyUSB` 和 `ttyACM` 分组执行相同的数值排序及两端交替，再合并、去重为完整
候选队列。排序只影响速度；所有候选最终都会被扫描，且仍只能由合法 MAVLink 帧确认
身份。明确路径模式不应用候选排序。

某个候选出现下列情况时关闭它并继续下一个：

- 打开失败或没有权限；
- 已被其他进程独占；
- 监听窗口内没有数据；
- 只收到普通文本、AT响应、诊断二进制或 CRC 错误帧。

任何通过官方 MAVLink CRC 校验的完整帧都能确认端口。探测所消费的第一帧必须连同
链路一起返回并交给正常业务解析，不能因自动发现丢失首帧。

一轮没有发现目标时，程序等待固定间隔后重新扫描，永不因为暂时缺少 STM32 串口而
退出。等待过程必须响应 `SIGINT`/`SIGTERM`，确保 `systemctl stop` 能及时终止程序。

## 7. 运行中断线恢复

发现成功后继续沿用现有端点学习规则：从持续收到且符合固件身份约定的 HEARTBEAT
学习 STM32 `sysid/compid`。如果发现首帧恰好是数据帧，先按正常业务处理，再等待随后
到达的 HEARTBEAT 完成端点学习。如果读取或写入返回 `EIO`、`ENODEV` 等不可恢复
错误，或者连续10秒没有收到任何合法 MAVLink 帧，则统一判定当前链路失效并执行：

1. 记录一次“STM32串口断开”错误；
2. 释放旧链路；
3. 清除从旧链路学习到的 STM32 sysid/compid；
4. 停止向失效链路发送5G状态和控制帧；
5. 返回串口发现流程；
6. 重新插入设备后自动发现新节点并恢复业务。

10秒是本阶段固定的静默超时值，不增加新的配置字段。固件会固定频率持续发送
HEARTBEAT 和数据帧，因此正常链路会持续刷新；即使USB驱动没有及时返回读错误，也能
在10秒内回到重新发现流程。

MQTT连接在串口重发现期间继续保持。此时收到下行控制命令应立即返回明确的
“STM32链路不可用”回执，不得退化为含义模糊的 MCU ACK 超时。

重新发现成功后，固件 sysid 仍按当前约定从 HEARTBEAT 学习，不写死设备间可能变化
的值。当前固件在 USART6 上统一使用 `compid=193`，但发现正确性的必要条件仍是合法
MAVLink 帧，不额外把该常量作为 USB 串口筛选条件。

## 8. 日志要求

日志使用已经独立出来的 `logging::Logger`，至少覆盖：

- 开始等待 STM32 MAVLink 串口；
- 候选设备无权限或被占用；
- 一轮扫描没有找到有效 MAVLink；
- 成功发现实际设备路径；
- 运行中串口断开；
- 连续10秒未收到合法 MAVLink 帧；
- 重新发现并恢复。

重复失败日志必须限频，例如等待状态最多每10秒输出一次汇总告警。不得按候选、按
读取轮次或按收到的垃圾字节刷屏。成功、断开和恢复是状态变化，不受等待告警限频影响。

日志示例：

```text
[WARN ] 尚未发现STM32 MAVLink串口，继续等待
[INFO ] 已发现STM32 MAVLink串口: /dev/ttyUSB6
[ERROR] STM32串口断开: /dev/ttyUSB6
[INFO ] 已重新发现STM32 MAVLink串口: /dev/ttyUSB0
```

## 9. 本地控制工具

`cns_control_test` 与主程序复用同一个串口发现模块：

- dry-run 行为保持不变，不读取配置、不枚举或打开硬件；
- 真实发送且 `serial.device="auto"` 时先自动发现 MAVLink 端口；
- 工具把发现期间收到的首帧交给现有解析逻辑，并继续等待固件固定频率发送的
  HEARTBEAT 来学习端点；随后执行现有的 COMMAND_LONG 发送和 COMMAND_ACK 配对；
- 主服务占用端口时，工具仍明确提示先停止 `cns-rpi`，不得绕过独占保护。

## 10. 自动化测试

自动化测试不依赖真实 USB 或 STM32，通过注入候选枚举、打开、读取和时间行为覆盖：

- 候选设备排序和去重；
- `ttyUSB`/`ttyACM` 尾号按整数自然排序，不出现 `1,10,2`；
- 奇数和偶数数量的候选都按最低、最高交替顺序完整输出且不遗漏；
- CRC 正确的 MAVLink 帧确认端口；
- 普通文本、垃圾字节和 CRC 错误帧不确认端口；
- 非 HEARTBEAT 的合法 MAVLink 数据帧也能确认端口；
- 探测期间收到的第一帧完整返回；
- 单个候选超时后继续下一个；
- 一轮失败后进入下一轮而非退出；
- 明确路径模式不枚举其他设备；
- 等待日志按时间限频；
- 无完整帧与设备断开使用不同结果；
- 首帧初始化静默计时，任意合法帧刷新计时，普通字节和CRC错误帧不刷新；
- 连续不足10秒不误判，达到10秒后判定链路失效；
- 链路失效后状态机回到发现状态并清除旧端点；
- 本地控制工具的 auto 模式复用相同发现规则；
- dry-run 仍完全不接触硬件。

## 11. 树莓派实机验收

1. CH340 为任意 `/dev/ttyUSBn` 时启动服务，确认自动发现成功。
2. 插拔其他 USB 串口使 CH340 枚举号改变，不修改配置仍能启动。
3. 确认 RM500U 的多个 AT/诊断端口不会被误认成 MAVLink 链路。
4. 运行中拔掉 CH340，确认服务进程不退出、日志明确且 MQTT继续连接。
5. 模拟设备节点仍存在但没有合法 MAVLink 的静默失联，确认10秒后进入重新发现。
6. 重新插入 CH340，确认服务自动发现新设备号并恢复遥测。
7. 等待至少一分钟，确认未发现状态的日志不会持续刷屏。
8. 停止主服务后运行 `cns_control_test`，确认自动发现、命令下发和 ACK 逻辑保持一致。
9. 主服务运行时执行真实控制工具，确认仍返回明确的串口占用诊断。

## 12. 不在本次范围内

- 不修改 STM32 固件发送逻辑或 MAVLink 帧格式。
- 不依赖或安装自定义 udev 规则。
- 不把自动发现结果持久化回 `config.json`。
- 不用 USB VID/PID、物理端口或固定 HEARTBEAT 身份替代 MAVLink CRC 确认。
- 不改变 RM500U 拨号服务的 AT 口发现逻辑。
