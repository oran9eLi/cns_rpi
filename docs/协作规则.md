# 仓库协作规则

版本：2026-07-01
状态：草案，待确认

## 1. 分支策略

- 默认分支：`main`，始终保持可编译、可运行的状态，不直接在 `main` 上做实验性修改。
- 日常开发在功能分支上进行，命名建议 `<type>/<简短描述>`，例如 `feat/uart-frame-reader`、`fix/mqtt-reconnect`。
- 功能分支完成并自测通过后合入 `main`（当前是两人协作规模，先不强制要求 PR + 代码评审，是否需要引入待你确认）。
- 不对已推送到 `main` 的提交做 `push --force` 或改写历史。

## 2. 提交信息规范

格式：

```text
<type>: <简短中文说明>
```

`type` 用小写英文，冒号后留一个空格，说明部分用简短中文描述本次提交的主要变化。

| type | 说明 |
|---|---|
| `feat` | 新增功能或能力接入 |
| `fix` | 修复问题 |
| `docs` | 文档更新 |
| `refactor` | 代码重构，不改变功能行为 |
| `chore` | 工程配置、仓库维护、脚本或杂项调整 |
| `build` | CMake、依赖、构建脚本调整 |
| `test` | 测试用例或验证记录调整 |

示例：

```text
feat: 接入UART帧同步与MAVLink解码
fix: 修复MQTT断线重连未清空发送队列
docs: 补充V1设计文档
chore: 初始化仓库忽略规则
```

提交说明保持简短明确，不写长篇背景。详细设计、验证结果和遗留问题记录在 `docs/` 对应文档或变更记录里，不塞进 commit message。

## 3. 代码注释规范

新增或重构的源码使用 Doxygen 风格中文注释，重点说明模块职责、数据含义、调用边界，不做逐行翻译。

- 每个 `.hpp`/`.cpp` 文件开头写文件头注释：职责、所属层级（对应设计文档里的 `uart/`/`protocol/`/`mqtt/` 等模块划分）、主要依赖、禁止事项。
- 每个公开类/函数在声明处写注释，说明入参、返回值、失败语义、调用线程/上下文限制（比如"只能在 comm 线程调用，不得在信号处理函数里调用"）。
- 结构体/字段涉及物理量、单位、时间戳、取值范围时必须写清楚，尤其是 MAVLink 解码出来的字段，要注明和固件端哪个消息/字段对应。
- 注释说明"为什么这样设计、谁调用、有什么限制"，不是复述代码本身。
- 修改函数行为、字段含义或模块边界时，必须同步更新注释；不写过期设计或"计划中但还没做"当成既成事实的描述。

文件头示例：

```cpp
/**
 * @file mavlink_frame_reader.hpp
 * @brief UART 字节流到完整 MAVLink 帧的同步与校验。
 *
 * @details
 * 本模块只负责帧同步和 CRC 校验，不解析消息语义。
 * 依赖边界：
 * - 只依赖 POSIX termios 和官方 mavlink/c_library_v2 头文件。
 * - 不包含 mqtt/、payload/ 等下游模块的头文件。
 */
```

## 4. 变更记录

架构、协议对接范围、身份策略等**影响后续维护判断**的变更，在 `docs/change_history/` 下新增一份日期编号的记录（例如 `2026-07-05_xx_UART链路对接.md`），至少包含：

- 变更摘要
- 影响范围（对应设计文档哪一节）
- 验证结果（怎么测的、结果如何）
- 遗留/未完成项

日常小改动（改个变量名、修个笔误）不需要单独记录，commit message 够用。

## 5. 仓库跟踪范围

本仓库跟踪：

- 源码（`src/`）
- CMake 工程文件
- 配置文件（`config/config.example.json`，真实密钥/密码不进仓库）
- `docs/` 下的设计文档、协作规则、变更记录
- `systemd/`、`scripts/` 等部署脚本

本仓库不跟踪（`.gitignore` 待建立时落实）：

- `build/` 等 CMake 构建产物
- 本地真实配置（如带 broker 密码的 `config.json`，只提交 `.example.json`）
- IDE 本地状态
- `.claude/` 本地助手状态

## 6. 构建验证

- 修改代码后本地跑一次 CMake 构建，开 `-Wall -Wextra`，警告需要清零或有明确理由说明为什么保留。
- 涉及 UART/MAVLink 解析、MQTT 收发的改动，跑一遍 `tests/` 下的单元测试（半帧、粘包、CRC 错误、断线重连这类边界场景，具体测试项等实现阶段再细化）。
- 无法在提交前完整验证的（比如依赖真机 STM32 联调），要在 commit message 或变更记录里写清楚"未验证的部分是什么"，不能含糊带过。

## 7. 文档同步要求

架构、目录结构、协议对接范围、身份策略发生变化时，必须同步更新 `docs/V1设计文档.md` 对应章节，不能只改代码不改文档。
