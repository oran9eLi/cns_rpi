# 新设备部署手册

从一台刚烧好系统的树莓派，部署到可交付的产品化状态。

目标平台：Raspberry Pi 5 / Raspberry Pi OS 64-bit（trixie / Debian 13）。

分两个阶段：**阶段 A** 让程序正常跑起来，**阶段 B** 做产品化加固（只读根文件系统）。阶段 A 结束时设备已经可用，阶段 B 可以另行安排。

---

## 阶段 A：基础部署

### A1. 准备用户与网络

产品化脚本假定用户名为 `dcdw`、仓库位于 `/home/dcdw/cns_rpi` —— `scripts/deploy.sh` 会显式校验这两点并拒绝在其他位置运行。烧录系统时就把用户建成 `dcdw`。

确认能 SSH 登录，并且 `dcdw` 具备 sudo 权限。

### A2. 克隆仓库

```bash
git clone https://ghfast.top/https://github.com/oran9eLi/cns_rpi.git ~/cns_rpi
cd ~/cns_rpi
```

首次克隆必须带 `ghfast.top` 镜像前缀：全局 git 镜像重写是 `install_deps.sh` 配置的，而脚本本身在仓库里，此时还没生效。

### A3. 安装依赖

```bash
./scripts/install_deps.sh
```

切换 apt 源到清华 TUNA、安装构建依赖、配置 git 全局镜像重写。跑完之后 remote 可以改回裸地址：

```bash
git remote set-url origin https://github.com/oran9eLi/cns_rpi.git
```

### A4. 创建现场配置

```bash
cp config/config.example.json config/config.json
```

按现场情况编辑 `config/config.json`，至少确认这几项：

| 字段 | 说明 |
|---|---|
| `identity.school_name` | 学校名称 |
| `mqtt.connection.host` / `port` | broker 地址。内网联调与正式公网入口的取舍见 `docs/V1设计文档.md` 第 9 节 |
| `serial.device` | **保持 `auto`**。CH340 的设备号会随重启变化，不要写死 `/dev/ttyUSBn` |
| `cellular.*` | 有物联卡时按 APN 配置 |

校验：

```bash
python3 -m json.tool config/config.json >/dev/null && echo OK
```

`config/config.json` 在 `.gitignore` 中，不会进仓库。

### A5. 部署

```bash
./scripts/deploy.sh
```

脚本会构建、安装配置 helper 与三个 systemd 单元、安装 journald 与 swap 配置，并把 `config/config.json` 迁移到 `/var/lib/cns-rpi/config.json`（原文件改名保留为 `.migrated`）。

### A6. 验证

```bash
systemctl is-active cns-rpi.service                     # active
journalctl -u cns-rpi.service -n 20 --no-pager          # 应出现"已发现STM32 MAVLink串口"
ctest --test-dir build                                  # 全部通过
```

没有 SIM 卡时 `cellular-dialup.service` 反复失败重试是预期现象，不影响主程序。

**到这里设备已经可用。** 若暂不做产品化加固，可以停在这里。

## 阶段 B：产品化加固（只读根文件系统）

### ⚠️ 开始前必读

**这一阶段必须能物理接触设备。** 要修改 `cmdline.txt` 并重启，一旦启动失败 SSH 立即失联，只能拔 SD 卡恢复。`cmdline.txt` 位于独立的 vfat 分区，插到任意一台电脑上就能改回来 —— 这是回退生命线。

先备份：

```bash
sudo cp /boot/firmware/cmdline.txt /boot/firmware/cmdline.txt.bak
```

**另外记住一条铁律**：overlay 一旦启用，根文件系统上的一切写入（包括 `git pull`、`deploy.sh`、任何 `apt install`）都只落在内存里，重启即蒸发，而过程看起来完全成功。以后所有维护都必须按下面第 B5 节的流程走。`deploy.sh` 已内置检测，overlay 生效时会直接拒绝运行。

### B1. 创建配置持久化卷

```bash
./scripts/cns-rpi-init-config-volume.sh
./scripts/deploy.sh          # 安装并启用挂载服务
```

创建 `/boot/firmware/cns-config.img`（16 MiB ext4），把现场配置迁入，并在根分区建好挂载点。

配置必须放在这个独立文件系统上，不能放在根分区：overlay 的 lowerdir 被内核钉住，写完之后无法 remount 回只读（EBUSY），断电保护会失效。原理见 `docs/OverlayFS只读根文件系统设计.md` 第 5 节。

确认：

```bash
grep 'cns-rpi' /proc/mounts        # /dev/loopN /var/lib/cns-rpi ext4 ro
```

### B2. 安装 overlayroot（第一次重启）

```bash
sudo apt-get install -y overlayroot
sudo reboot
```

装包**不会**启用 overlay（默认 `overlayroot=""`），这一步风险为零，单独重启一次是为了确认 initramfs 重新生成后系统仍能正常启动。

重启后确认 SSH 可用、`cns-rpi.service` 为 `active`。

### B3. 启用 overlay（第二次重启）

```bash
sudo systemctl mask systemd-remount-fs.service
sudo raspi-config nonint enable_overlayfs
sudo reboot
```

`mask systemd-remount-fs` 是**必须的配套动作**，不是可选项：overlayfs 不支持 reconfigure，该单元每次开机必然失败，会拖垮依赖它的整条链（实测中 swap 就是这样完全失效的）。撤销 overlay 时必须同时 `unmask`，理由见 `docs/OverlayFS只读根文件系统设计.md` 第 7 节。

### B4. 验证

```bash
findmnt -n -o FSTYPE /                             # overlay
grep ' /media/root-ro ' /proc/mounts               # ro
grep 'cns-rpi' /proc/mounts                        # /var/lib/cns-rpi ext4 ro
systemctl is-active cns-rpi-config.service cns-rpi.service
cat /proc/swaps                                    # /dev/zram0
systemctl --failed                                 # 0 个
python3 -c 'import json;print(json.load(open("/var/lib/cns-rpi/config.json"))["runtime"])'
```

判据：

- 根挂载类型为 `overlay`。只看 `cmdline.txt` 内容不算验证；
- **失败单元必须为 0**。出现 `systemd-remount-fs.service` 说明 B3 的 mask 被漏了；
- **配置内容与重启前一致**。这是持久化闭环唯一的最终证明。

### B5. 此后的维护流程

overlay 启用后，任何代码更新或系统改动都必须这样做：

```bash
sudo raspi-config nonint disable_overlayfs && sudo reboot
# 重启后
cd ~/cns_rpi && git pull && ./scripts/deploy.sh
sudo raspi-config nonint enable_overlayfs && sudo reboot
```

单个配置文件的临时修改可以用 `sudo overlayroot-chroot`，但不要用它跑完整部署 —— 构建产物、服务状态和挂载关系都不在 chroot 视图内。

## 常见问题

**主程序起不来，日志说读不到配置**

检查配置卷是否挂上：`systemctl status cns-rpi-config.service`。挂载失败时 `cns-rpi.service` 会因 `Requires=` 而拒绝启动 —— 这是有意设计，避免它读到挂载点下的空目录、把缺省配置当成现场配置跑起来。

**改了东西，重启后全没了**

overlay 还开着。按 B5 的流程重做。

**`deploy.sh` 报"检测到 OverlayFS 已启用"**

正常拦截，按 B5 处理。这道拦截是为了防止"部署全部成功、重启全部蒸发"的静默故障。

**串口找不到 / 设备号变了**

`serial.device` 应为 `auto`，程序按合法 MAVLink 帧自动发现，不依赖固定设备号。仍找不到时检查 CH340 连接、STM32 供电与固件是否在发心跳，不要改回固定设备号。
