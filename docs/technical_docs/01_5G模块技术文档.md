## 1. 模块

5G 模块为 Raspberry Pi 5 提供蜂窝 IP 网络接入，使树莓派在现场没有有线网络或 Wi-Fi 时仍可连接外部 IP 网络。本工程实装的器件是 **Quectel RM500U-CNV**，不是产品设计书建议的 RG660Qx，也不将其描述为 5G-A 模块。

模块通过 USB 与树莓派连接：USB 串口用于发送 AT 指令，NCM 网卡 `usb0` 用于承载 IP 数据。自动拨号由独立 Python 脚本和 systemd 单元完成，与 `cns_rpi` C++ 主程序、STM32 UART/MAVLink 链路相互独立。

正式接入链路：

```text
Quectel RM500U-CNV
  -> USB 复合设备（USB ID 2c7c:0900）
  -> /dev/ttyUSB*（按 USB interface 05 动态识别主 AT 口）
  -> scripts/cellular_dialup.py
       1. 查询/设置 PDP 上下文
       2. 激活 PDP
       3. 将 PDP 绑定到 NCM 网卡
  -> usb0（LOWER_UP）
  -> NetworkManager（DHCP/SLAAC 与路由）
  -> IP 网络
```

本文只覆盖上述 5G 接入链路，不展开运行在 IP 网络之上的 MQTT 遥测、服务器协议和 STM32 数据链路。

## 3. 硬件与总线

| 项目 | 实际配置 |
|---|---|
| 器件 | Quectel RM500U-CNV |
| 实测固件版本 | `RM500UCNVAAR03A14M2G` |
| 主机 | Raspberry Pi 5，Raspberry Pi OS 64-bit |
| 物理接口 | USB 复合设备 |
| USB ID | `2c7c:0900` |
| 主 AT 口 | USB interface number `05`；节点编号不固定 |
| AT 串口参数 | 115200 bit/s，pyserial 默认 8N1，读超时 1 s |
| 数据接口 | NCM 网卡 `usb0`，netcard index `0` |
| 当前 SIM | 普通中国移动手机 SIM，未启用 PIN 锁 |
| 当前 APN/PDP | `cmnet`，CID `1`，PDP 类型 `IPV4V6` |

RM500U-CNV 本次枚举出 5 个 `/dev/ttyUSB*` 端口。设备节点编号会随其他 USB 设备插拔而变化，因此程序不写死 `/dev/ttyUSB3`，而是对 `/dev/ttyUSB*` 执行 `udevadm info`，按 `ID_USB_INTERFACE_NUM=05` 识别主 AT 口。

实测 USB interface 映射如下，仅作为诊断参考：

| USB interface | 实测用途 |
|---|---|
| `02` | 未测试 |
| `03` | 诊断/日志口，不是 AT 口 |
| `04` | 可接收 AT，但不是主 AT 口 |
| `05` | 主 AT 指令口 |
| `06` | 未测试 |

## 4. 软件分层与依赖

| 层/组件 | 文件或系统服务 | 职责 |
|---|---|---|
| 配置 | `config/config.example.json` 的 `cellular` 节 | 提供 APN、CID、AT 口 USB interface number 和枚举等待时间 |
| 拨号脚本 | `scripts/cellular_dialup.py` | 动态识别 AT 口、发送和解析 AT 命令、激活 PDP、绑定 NCM、确认 `usb0` 载波 |
| 启动与重试 | `systemd/cellular-dialup.service` | 开机执行一次拨号；失败后每 15 s 重新启动脚本 |
| 网卡配置 | NetworkManager | 在 `usb0` 出现载波后完成 DHCP/SLAAC 和路由配置 |
| 依赖安装 | `scripts/install_deps.sh` | 安装 `python3-serial` 等运行和构建依赖 |
| 单元测试 | `scripts/test_cellular_dialup.py` | 验证配置读取、AT 响应解析和纯逻辑判断，不访问真实硬件 |

边界约束：

- 拨号脚本是独立 Python 进程，不并入 CMake，也不复用 C++ 的 `uart::SerialPort`。
- 脚本只读取 `config.json` 的 `cellular` 节，不读取 MQTT、STM32 串口或业务数据配置。
- 脚本不执行 DHCP、不修改路由表；这些职责属于 NetworkManager。
- 脚本内部只执行一轮有边界的拨号流程；跨轮重试由 systemd 负责。

## 5. 数据结构与契约

本模块没有向主程序发布 C/C++ 数据结构，接口契约由配置、AT 文本协议、进程退出码和 Linux 网卡状态组成。

### 5.1 配置契约

```json
"cellular": {
  "apn": "cmnet",
  "cid": 1,
  "usb_interface_number": "05",
  "at_port_wait_seconds": 30
}
```

| 字段 | 类型 | 当前示例值 | 说明 |
|---|---|---|---|
| `apn` | 字符串 | `cmnet` | 当前手机 SIM 的 APN；更换正式物联卡后按运营商资料修改 |
| `cid` | 整数 | `1` | 用于上网的 PDP context ID |
| `usb_interface_number` | 字符串 | `05` | 主 AT 口的 USB interface number；保留前导零 |
| `at_port_wait_seconds` | 整数 | `30` | 等待 AT 口枚举的最长时间，单位为秒 |

配置缺失、字段类型错误或文件不可读时，脚本打印错误并以非零状态退出。

### 5.2 AT 与状态契约

| 契约 | 含义 |
|---|---|
| AT 命令结束符 | `\r\n` |
| 单条 AT 命令超时 | 5 s |
| 成功终止行 | `OK` |
| 失败终止行 | `ERROR` 或 `+CME ERROR...` |
| 脚本退出码 `0` | 拨号序列完成，且 `ip link show usb0` 已出现 `LOWER_UP` |
| 脚本退出码非 `0` | 配置、设备枚举、串口、AT 命令或载波确认失败 |

退出码 `0` 不等价于已经完成公网业务请求；IP 地址分配、DNS 和默认路由仍由 NetworkManager 负责。

## 6. 拨号 / 处理流程

1. systemd 在开机阶段启动 `scripts/cellular_dialup.py <config.json>`。
2. 脚本每 1 s 扫描一次 `/dev/ttyUSB*`，通过 `udevadm` 匹配配置的 USB interface number；超过 `at_port_wait_seconds` 仍未找到则失败退出。
3. 以 115200 bit/s 打开主 AT 串口。
4. 发送 `AT+CGDCONT?`，查询目标 CID 对应的 PDP 上下文。
5. 若目标 CID 不存在或 APN 与配置不一致，发送：

   ```text
   AT+CGDCONT=<cid>,"IPV4V6","<apn>"
   ```

6. 发送 `AT+CGACT=1,<cid>` 激活 PDP。若命令返回错误，再用 `AT+CGACT?` 查询实际状态；目标 CID 已激活时继续，否则失败退出。
7. 发送 `AT+QNETDEVCTL=<cid>,1,0`，将 PDP 上下文绑定到 NCM 网卡 0。
8. 最多等待 10 s，轮询 `ip link show usb0`；看到 `LOWER_UP` 后成功退出。
9. NetworkManager 检测到 `usb0` 载波后自动完成 IPv4 DHCP、IPv6 SLAAC 和路由配置。
10. 任一步失败时脚本非零退出；systemd 等待 15 s 后重新运行整套流程。

本流程是在 `AT+QCFG="usbnet"` 返回模式 `5`（NCM）的实机环境中验证的，不应直接套用于 MBIM、QMI 等其他 USB 网络模式。

## 7. 接口清单

### 7.1 脚本入口

```bash
python3 scripts/cellular_dialup.py config/config.json
```

参数必须是包含 `cellular` 节的 JSON 配置文件路径。成功返回 `0`，失败返回 `1`；参数数量错误返回 `2`。

### 7.2 主要内部接口

| 接口 | 职责 |
|---|---|
| `load_cellular_config(path)` | 读取并转换 `cellular` 配置 |
| `find_at_port(interface_number, wait_seconds)` | 动态扫描并返回匹配的 AT 设备节点 |
| `send_at_command(ser, command, timeout)` | 发送一条 AT 命令并等待终止行 |
| `parse_cgdcont_apn(lines, cid)` | 从 `AT+CGDCONT?` 响应解析指定 CID 的 APN |
| `parse_cgact_active(lines, cid)` | 从 `AT+CGACT?` 响应判断指定 CID 是否激活 |
| `wait_for_carrier(iface, timeout_seconds)` | 等待 Linux 网卡出现 `LOWER_UP` |
| `dial_up(config)` | 编排一次完整拨号流程，返回布尔结果 |

### 7.3 systemd 入口

`systemd/cellular-dialup.service` 当前按真机部署路径执行：

```text
/usr/bin/python3 /home/dcdw/cns_rpi/scripts/cellular_dialup.py /home/dcdw/cns_rpi/config/config.json
```

若部署用户或仓库路径变化，安装 service 前必须同步修改 `ExecStart`。

## 8. 配置项

| 配置位置 | 关键项 | 当前行为 |
|---|---|---|
| `config/config.example.json` | `cellular.apn` | 决定查询不匹配时写入的 APN |
| `config/config.example.json` | `cellular.cid` | 决定激活和绑定的 PDP context ID |
| `config/config.example.json` | `cellular.usb_interface_number` | 决定选择哪个 `/dev/ttyUSB*` 作为主 AT 口 |
| `config/config.example.json` | `cellular.at_port_wait_seconds` | 决定单次脚本等待 USB 枚举的时限 |
| `scripts/cellular_dialup.py` | `AT_BAUDRATE = 115200` | AT 串口速率 |
| `scripts/cellular_dialup.py` | `AT_COMMAND_TIMEOUT_SECONDS = 5` | 单条 AT 命令超时 |
| `scripts/cellular_dialup.py` | `CARRIER_WAIT_SECONDS = 10` | 等待 `usb0` 载波超时 |
| `scripts/cellular_dialup.py` | `NCM_NETCARD_INDEX = 0` | NCM 网卡序号 |
| `scripts/cellular_dialup.py` | `NCM_IFACE_NAME = "usb0"` | Linux 网卡名称 |
| `systemd/cellular-dialup.service` | `RestartSec=15` | 单次拨号失败后的重试间隔 |

当前代码没有独立的 5G 启用/禁用配置项。是否开机自动拨号由 systemd 单元是否安装并 `enable` 决定。

## 9. 异常与恢复

| 异常 | 当前处理 |
|---|---|
| 配置文件不可读、缺字段或类型错误 | 打印错误，非零退出 |
| AT 口未在时限内枚举 | 打印超时与目标 interface number，非零退出 |
| `udevadm` 查询单个节点失败 | 跳过该节点，继续扫描 |
| 主 AT 口打开失败 | 打印设备节点和异常，非零退出 |
| AT 命令超时、返回 `ERROR` / `+CME ERROR` | 当前步骤失败并非零退出；`CGACT` 例外地先查询实际激活状态 |
| `usb0` 未在 10 s 内出现 `LOWER_UP` | 打印超时，非零退出 |
| 单次拨号失败 | systemd 在 15 s 后重新启动脚本 |
| 拨号成功后发生运行期断链 | 当前 oneshot 服务已退出，不持续监测，也不会因此自动触发重拨 |
| 模块状态异常需要平台报警 | 当前未接入 `cns_rpi` 状态或报警数据 |

脚本不会因 5G 拨号失败阻塞 STM32 UART/MAVLink 链路。若后续需要运行期链路守护、信号质量上报或平台报警，应另行设计常驻监测机制。

## 10. 调试与验证

### 10.1 已完成的实机验证

实机记录见 `docs/5G模块拨号验证.md`。在普通中国移动手机 SIM、APN `cmnet` 条件下已确认：

- SIM 状态 `READY`，模块已附着分组域。
- `AT+QNWINFO` 返回 `NR5G-SA`，实测注册到 5G SA 网络。
- `AT+CGACT=1,1` 可激活 CID 1；仅激活 PDP 时 `usb0` 仍可能无载波。
- `AT+QNETDEVCTL=1,1,0` 后 `usb0` 进入 `LOWER_UP`。
- NetworkManager 可自动获取 IPv4 地址、IPv6 地址和默认路由。
- 通过 `usb0` 的 DNS 与公网连通性已经人工验证。

上述结果证明当前模块、当前固件、当前 SIM 和 NCM 配置下的拨号链路可用，不代表任何环境下的固定吞吐率或时延保证。

### 10.2 自动化测试

```bash
cd scripts
python3 -m unittest test_cellular_dialup -v
```

单元测试覆盖配置读取、PDP/APN 响应解析、PDP 激活状态解析以及 AT 成功、错误、CME 错误和超时处理。动态 USB 枚举、真实串口、AT 模块响应、`usb0` 载波和公网连通性必须在实机验证。

### 10.3 现场检查命令

```bash
lsusb
udevadm info -q property -n /dev/ttyUSB3
systemctl status cellular-dialup.service
ip link show usb0
ip address show usb0
ip route
journalctl -u cellular-dialup.service
```

`/dev/ttyUSB3` 只是在已记录实机上的示例。现场应先根据 `ID_USB_INTERFACE_NUM` 确认设备节点，不能照抄固定编号。
