# 5G 模块（Quectel RM500U-CNV）拨号验证记录

日期：2026-07-08
性质：临时真机验证记录，不是正式设计文档——为以后 M2+ 设计"开机自动拨号"的 systemd
单元时提供已验证过的事实依据，避免到时候凭 Quectel 通用文档瞎猜命令。

## 测试前提（重要，会过期）

**这次插的是一张普通手机 SIM 卡，不是物联卡。** 下面记录里 `AT+CGDCONT?` 查到的默认
APN 是 `cmnet`（中国移动手机卡的标准上网 APN），**这是这张卡自带的，不是我们手动配的**。
以后换成正式的物联卡，APN 大概率会变（物联网卡通常用运营商单独分配的专用 APN，不是
`cmnet`），到时候：

- 如果新 APN 卡本身预置了对应的 `CGDCONT` 上下文（像这次一样开箱即用），流程不变，
  只是 cid/APN 名字换了，需要重新跑一遍 `AT+CGDCONT?` 确认。
- 如果物联卡没有预置上下文，需要先用 `AT+CGDCONT=<cid>,"IPV4V6","<物联卡APN>"` 手动定义，
  这条命令这次没有实际测过（这次是现成的），到时候要单独验证一次。

其余拨号步骤（PDP 激活、NCM 绑定、DHCP）跟 APN 是哪个无关，理论上换卡后不用改。

## 正式物联卡补充验证（2026-07-21）

树莓派更换正式中国移动物联卡后，停止 `cellular-dialup.service`
并通过 USB interface number `05` 的主 AT 口查询，得到：

```text
AT+CPIN?        -> +CPIN: READY
AT+COPS?        -> +COPS: 0,0,"China Mobile",11
AT+CGCONTRDP=1  -> +CGCONTRDP: 1,5,CMMTM5GDCDW.JS,...
AT+CGACT?       -> +CGACT: 1,1
```

因此正式物联卡的专用 APN 为 `CMMTM5GDCDW.JS`，仓库默认配置已从
`cmnet` 更新为该值。`AT+CGDCONT?` 当时仍显示旧配置 `cmnet`，这反映的是
模块当前 PDP 配置；`AT+CGCONTRDP=1` 返回的是已激活上下文的网络动态参数。
使用新配置重新执行完整拨号流程后，还需再确认 `usb0` 载波、地址和路由。

## 硬件识别

模块：Quectel RM500U-CNV，USB ID `2c7c:0900`，固件版本 `RM500UCNVAAR03A14M2G`。

暴露 5 个 `/dev/ttyUSB*` 端口，**端口号会随其它 USB 设备的插拔而变化，不能死记
`ttyUSBn` 这个数字**，要靠 USB interface number 识别（这次实测映射，供参考）：

| USB interface | 用途 | 这次对应的设备节点 |
|---|---|---|
| 02 | 未测试 | `ttyUSB0` |
| 03 | 诊断/日志口（持续吐二进制 NV/log 数据，不是 AT 口） | `ttyUSB1` |
| 04 | 能收 AT 但回 `+CME ERROR: 4`，不是主 AT 口 | `ttyUSB2` |
| **05** | **主 AT 指令口（`ATI`/`AT+CPIN?`等都在这条上）** | `ttyUSB3` |
| 06 | 未测试 | `ttyUSB4` |

判断方法：`udevadm info -q property -n /dev/ttyUSBx | grep ID_USB_INTERFACE_NUM`。

## 验证过程与结果

### 1. SIM/信号/注册状态

```
AT+CPIN?        -> +CPIN: READY                          （SIM已识别，无需PIN）
AT+CSQ          -> +CSQ: 19,99                            （信号尚可）
AT+COPS?        -> +COPS: 0,0,"CHINA MOBILE",11
AT+QNWINFO      -> +QNWINFO: "NR5G-SA",46000,"NR N28",152650  （已注册5G SA）
AT+QCCID        -> +QCCID: 898600401024F5249386           （ICCID，确认是张真实SIM）
AT+CFUN?        -> +CFUN: 1                               （射频功能已开启）
AT+CGATT?       -> +CGATT: 1                              （已附着分组域）
```

`AT+CREG?`/`AT+CGREG?` 返回 `0,0`（未注册/未搜索）——这个不代表有问题，NR5G-SA 是纯数据网络，
没有 CS 域，`CGATT=1` 已经确认分组域附着成功，是更准确的判断依据。

### 2. 默认 PDP 上下文（这次是手机卡自带的，见上面"测试前提"）

```
AT+CGDCONT?
+CGDCONT: 1,"IPV4V6","cmnet","0.0.0.0",0,0,0,0,...
+CGDCONT: 11,"IPV4V6","ims","0.0.0.0",0,0,0,2,...
```

cid=1 是上网用的 `cmnet`，cid=11 是 IMS 信令用的（跟打电话/短信相关，不是拿来跑公网流量的）。

### 3. 拨号（激活 PDP + 绑定 NCM 网卡）——两步缺一不可

这是这次验证最关键的发现：**只激活 PDP 上下文拿到 IP 是不够的，`usb0` 网卡本身还是
`NO-CARRIER`**，必须再补一条 Quectel 专用命令才能让 `usb0` 真正起来：

```bash
AT+CGACT=1,1        # 第一步：激活cid=1(cmnet)的PDP上下文
                     # 效果：AT+CGPADDR能查到分配的IP了(比如10.52.55.91)
                     # 但此时 ip link show usb0 仍然是 NO-CARRIER！

AT+QNETDEVCTL=1,1,0  # 第二步：把PDP上下文1绑定到NCM网卡0（关键命令）
                     # 效果：usb0 立刻变成 LOWER_UP / state UP
```

命令格式：`AT+QNETDEVCTL=<pdp_cid>,<enable 1/0>,<netcard_index>`。这次用的是
`AT+QCFG="usbnet"` 查到的模式 `5`（NCM）下验证的，其它 USB 模式（比如 MBIM）不适用这条命令。

### 4. DHCP / 路由——不用自己写脚本

树莓派上 `NetworkManager` 本来就在跑（`Wired connection 2` 这个自动生成的以太网型配置），
`usb0` 一出现 `LOWER_UP` 载波，NetworkManager 自动就把它当一张普通有线网卡去 DHCP，
不需要手动跑 `dhclient`/`udhcpc`（这台机器上甚至没装 `dhclient`）。

拿到的地址：

```
usb0: inet 10.180.21.17/24 (dhcp)
      inet6 2409:8924:e86:7c2:.../64 (SLAAC)
default via 10.180.21.1 dev usb0 metric 100   <- 5G
default via 192.168.10.1 dev wlan0 metric 600 <- WiFi
```

**⚠️ 需要产品侧决定的问题**：`usb0` 的默认路由 metric（100）比 `wlan0`（600）小，
Linux 路由选择 metric 越小优先级越高，也就是说**现在实际出网走的是 5G，不是 WiFi**——
这是 NetworkManager 按谁先 DHCP 到谁优先的默认行为自动决定的，不是我们特意配置的。
如果最终产品希望 WiFi 优先、5G 仅作为断网备份，需要显式调整 metric 或连接优先级
（比如给 WiFi 连接固定更低的 metric，或者用 NetworkManager 的 connection priority）；
如果就是希望 5G 优先也可以维持现状，但这是一个需要显式确认的产品决策，不应该让
"谁先连上"这种偶然顺序来决定。

### 5. 公网连通性测试（全部走 `usb0`，`ping -I usb0`）

| 目标 | 结果 |
|---|---|
| `8.8.8.8`（Google DNS） | 4/4 通，RTT 72~523ms |
| `223.5.5.5`（阿里DNS） | 4/4 通，RTT 25~163ms |
| `baidu.com`（先 DNS 解析后 ping） | 域名正常解析出4个IP，ping通，RTT 90~160ms |
| `114.114.114.114`（国内公共DNS） | 100% 丢包（重测两次结果一致） |

`114.114.114.114` 不通大概率是这个 IP 本身不响应 ICMP（防扫描策略），不是我们这边网络
有问题——同一时间段其它域名/IP 全部正常，排除了链路本身的问题。

延迟波动比较大（同一目标最快最慢能差 5~6 倍），是真实蜂窝网络的正常现象，不代表异常，
但如果以后有对延迟敏感的功能（比如实时性要求高的指令下发），这个波动幅度需要纳入考虑。

## 结论 / 后续设计要点

1. 拨号本身**验证通过**：SIM 卡（哪怕是手机卡）、信号、5G SA 注册、拨号、DHCP、公网连通
   全链路走通了，NCM 模式下不需要 ModemManager/QMI，用这两条 AT 命令 + NetworkManager
   自动 DHCP 就够。
2. 换成物联卡后，"测试前提"里提到的 APN 差异需要重新跑一次本文档第1/2节的验证，
   其余步骤（`AT+CGACT`/`AT+QNETDEVCTL`）预期不用改。
3. 路由 metric 优先级是一个**待产品决策**的点，见第4节，不是技术限制，需要显式确认后再
   写进开机自动拨号的 systemd 单元设计里。
4. 下一步（M2+ 正式设计时）：把第3节这两条 AT 命令封装进一个开机执行一次的轻量脚本
   （或者 systemd oneshot unit），排在 `cns_rpi` 主程序之前/并行都可以（拨号失败不应该
   阻塞主程序启动，5G 只是可选的备用/主用上网通道，不是 MAVLink/UART 链路的依赖）。
