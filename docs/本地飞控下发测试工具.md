# 本地飞控下发测试工具

`cns_control_test` 用于在树莓派本地联调树莓派与 STM32 之间的飞控命令链路。它可以绕过 MQTT 和服务器路由，但不是生产环境的控制入口。

## 1. 数据流向

操作者在命令行或文件中提供的 JSON 只是树莓派本地输入格式。工具会复用主程序的命令解析和编码逻辑，将 JSON 转换为 MAVLink `COMMAND_LONG`，再序列化成 MAVLink 二进制帧通过 UART 发送给 STM32。

```text
本地 JSON 输入
  → 命令解析与校验
  → MAVLink COMMAND_LONG
  → UART 二进制帧
  → STM32 执行
  → MAVLink COMMAND_ACK
  → 本地 JSON 回执
```

UART 上不会传输 JSON。

## 2. 安全要求

> **警告：**执行电机 PWM、起飞、降落或急停的真实下发测试前，必须拆除桨叶或断开电机动力。

正式主程序和本工具不能同时占用 STM32 串口。真实发送前先停止主服务：

```bash
sudo systemctl stop cns-rpi.service
```

联调结束后再恢复主服务：

```bash
sudo systemctl start cns-rpi.service
```

本工具每次运行只处理一条命令，不提供跨进程幂等保护。重复执行同一条 `--send` 命令会向 STM32 重复发送，即使 `command_id` 相同也会再次执行。

## 3. 构建

```bash
cmake -S . -B build
cmake --build build --target cns_control_test -j2
```

## 4. 默认空运行

不带 `--send` 时默认为空运行。空运行只校验 JSON 并展示将要生成的 MAVLink 映射，不读取配置文件，不打开 UART，也不向 STM32 发送数据。

```bash
./build/cns_control_test config/config.json \
  '{"command_id":"local-pwm-001","command":"set_motor_pwm","parameters":{"pwm_us":[1500,1500,1500,1500]}}'
```

在命令行中建议用单引号包住整段 JSON，避免内部双引号被解析。空运行的 `target_system` 为 `null`，因为此时尚未从 STM32 HEARTBEAT 学习真实 `sysid`。

## 5. 从文件读取命令

命令文件示例 `control.json`：

```json
{
  "command_id": "local-takeoff-001",
  "command": "takeoff",
  "parameters": {}
}
```

默认空运行：

```bash
./build/cns_control_test --file control.json config/config.json
```

真实发送：

```bash
./build/cns_control_test --file control.json --send config/config.json
```

直接 JSON 和 `--file` 不能同时使用。

## 6. 真实发送

只有显式指定 `--send` 才会读取 `config.json` 中的串口配置、打开 UART 并发送 MAVLink 二进制帧。

四路 PWM：

```bash
./build/cns_control_test --send config/config.json \
  '{"command_id":"local-pwm-001","command":"set_motor_pwm","parameters":{"pwm_us":[1500,1500,1500,1500]}}'
```

起飞：

```bash
./build/cns_control_test --send config/config.json \
  '{"command_id":"local-takeoff-001","command":"takeoff","parameters":{}}'
```

降落：

```bash
./build/cns_control_test --send config/config.json \
  '{"command_id":"local-land-001","command":"land","parameters":{}}'
```

急停：

```bash
./build/cns_control_test --send config/config.json \
  '{"command_id":"local-stop-001","command":"emergency_stop","parameters":{}}'
```

工具打开串口后最多等待 5 秒学习 STM32 HEARTBEAT 中的动态 `sysid`。超时时不会发送控制命令。命令成功发送后，等待 STM32 `COMMAND_ACK` 的超时时间为 2 秒。收到 `MAV_RESULT_IN_PROGRESS` 时会输出进度回执并刷新等待超时，然后继续等待最终回执。

树莓派不使用固定 `sysid`。学习成功后，`COMMAND_LONG` 的来源为
“动态 `sysid` / `compid=191`”，目标为“同一动态 `sysid` /
`compid=193`”。

## 7. 输出约定

- stdout 只输出 JSON，每条回执独占一行，便于脚本逐行解析。
- stderr 输出参数、配置、串口、端点学习和故障诊断信息。
- 如果 STM32 先返回一次或多次 `MAV_RESULT_IN_PROGRESS`，stdout 会先输出对应的进度 JSON，再输出最终 JSON，因此真实发送不保证 stdout 只有一行。

退出码：

| 退出码 | 含义 |
|---:|---|
| `0` | 空运行成功，或 STM32 最终回执为 `accepted` |
| `1` | 控制 JSON 被拒绝、STM32 拒绝、ACK 超时，或未获得有效最终回执 |
| `2` | CLI 用法错误、命令文件读取失败，或配置读取失败 |
| `3` | UART 打开/写入失败（包括串口被其他进程占用），或等待 STM32 HEARTBEAT 超时 |

## 8. 故障排查

- 提示串口已被占用：先执行 `systemctl stop cns-rpi.service`，再重新运行工具。
- 提示其他串口打开失败：确认 `config.json` 中 `serial.device` 正确，并检查设备节点和用户权限。
- HEARTBEAT 超时：检查 STM32 供电、UART TX/RX 连接、波特率，以及 STM32 在 USART6 上是否发送 `type=18`、`compid=193` 的 HEARTBEAT。
- ACK 超时：检查 STM32 是否收到 `COMMAND_LONG`，并是否按请求来源回填 `COMMAND_ACK.target_system/target_component`。
- 命令被拒绝：检查命令名、`command_id`、四路 `pwm_us` 数量及 `1000..2000` 微秒范围，并查看 stdout 的 JSON 错误码和 stderr 诊断。

## 9. 2026-07-18 实机验证

在 Raspberry Pi 5 上停止主服务、保持 STM32 在线且断开电机后执行四路 PWM 真实下发：

```bash
./build/cns_control_test --send config/config.json \
  '{"command_id":"local-motor-disconnected-20260718-01","command":"set_motor_pwm","parameters":{"pwm_us":[1500,1500,1500,1500]}}'
```

工具通过自动发现学习到 STM32 `sysid=1, compid=193`，收到固件返回的 `COMMAND_ACK`，最终回执为：

```json
{"command":"set_motor_pwm","command_id":"local-motor-disconnected-20260718-01","mavlink_command":31013,"message":"单片机拒绝或执行命令失败","result":1,"result_code":"temporarily_rejected","status":"rejected"}
```

`result=1` 对应 `MAV_RESULT_TEMPORARILY_REJECTED`。该结果符合电机断开时固件拒绝执行的预期，同时证明 JSON 解析、`COMMAND_LONG` 编码与发送、动态端点学习、`COMMAND_ACK` 配对和 JSON 回执链路已经闭环；此前出现的 `mcu_ack_timeout` 在当前固件上未复现。
