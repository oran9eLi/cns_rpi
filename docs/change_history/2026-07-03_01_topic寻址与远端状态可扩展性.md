# 变更记录：Topic 寻址与远端状态可扩展性设计

日期：2026-07-03

## 变更摘要

用户提出 V2 构想：让一台 RPi 设备订阅并控制另一台 RPi 设备（一台当另一台的"管控中心"），实现方式是基于 MQTT topic 的发布/订阅——这在现有架构下可行，因为 RPi 的命令分发逻辑（`command_source` 接口）本身不关心命令的真正发布者是谁，只要发到硬件部服务器 broker 上对应设备的命令 topic 即可。

该功能本身明确留到 V2 实现，但用户要求 V1 架构现在就要为它预留扩展性，不能等 V2 再推倒重来。经讨论确定以下几处 V1 阶段现在就要锁定的设计决策（不写代码，只改设计文档）：

1. **Topic 设备寻址字段必须用厂商唯一产品识别码，不能用 V1 过渡期的 RPi 硬件序列号**——否则 V2 切换身份来源时 topic 名字必然变化，所有订阅方都要跟着重新订阅，这正是要避免的"以后重构"。
2. **由此带来的启动时序依赖**：厂商唯一 ID 要从 `OPEN_DRONE_ID_BASIC_ID` 帧解码才能拿到（M3c 范围），MQTT client 初始化因此必须等"身份就绪"之后才能进行，不能像之前设想的"读完配置就直接连"。
3. **Topic 命名方案的定稿时机相应后移**：不是到 M5 才第一次考虑，而是要等 M3c 完成之后才能真正拍板；不需要因此调整现有里程碑顺序（M3c 本来就排在 M5 之前）。
4. **新增 `remote_state/` 预留位**（与 `state/` 对称但数据来源不同）：V1 只在目录结构里记录职责边界，不建目录、不写代码，留给 V2 具体设计。
5. **"控制别的设备"不需要新模块**：直接复用现有 `mqtt/mqtt_client.hpp` 的 publish 能力，不经过本机 `command_dispatcher`/UART 转发那套逻辑（那套是"RPi → 自己的 STM32"专用链路）。

## 影响范围

对应 `docs/V1设计文档.md`：
- §2 数据流架构——新增启动时序依赖说明（点 2）
- §8 目录结构——新增 `remote_state/` 预留位说明（点 4），不改变 V1 实际会创建的目录树
- §9 MQTT 收发设计——新增 topic 寻址字段约束（点 1、3）和"控制别的设备不需要新模块"说明（点 5）
- §11 开放问题——新增两条待办：topic 命名方案最终格式待 M3c 后定稿；`remote_state/` 具体接口形状留到 V2

不涉及代码改动（当前仓库处于 M2 之后、M3 之前，MQTT/身份帧解码相关代码都还未实现）。

## 验证结果

本次是纯架构讨论（superpowers:brainstorming 流程），未涉及代码，无需运行构建/测试验证。设计决策通过与用户逐项确认（每个关键取舍点单独提问、用户选择后才写入文档），不是单方面假设。

## 遗留/未完成项

- Topic 命名方案的具体字符串格式（前缀、分段规则）——留到 M5 实现阶段，且必须等 M3c 完成、能拿到厂商唯一产品识别码后再定稿
- MQTT 初始化"等身份就绪"这一步的具体等待/超时/重试策略——留到 M5 实现阶段设计，本次只确定了"必须等"这个依赖关系，没有设计具体机制
- `remote_state/` 的接口形状、JSON payload 解码方式、与 V1 `state/`/`payload/` 模块的关系——留到 V2 立项时详细设计，V1 不做
